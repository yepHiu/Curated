## 背景与动机

### 当前处境

当前项目（Curated / jav-shadcn）是一个功能完善的桌面媒体库应用，但在以下方面存在局限：

- **内容敏感性**: 领域特定（成人内容），不适合在个人简历或公开场合展示
- **受众狭窄**: 无法作为技术能力展示的通用案例
- **职业风险**: 关联 NSFW 内容可能对职业发展产生负面影响
- **协作限制**: 难以邀请他人参与或对外分享

### 转型价值

将核心技术抽象并重构为**通用媒体库管理工具**带来的收益：

| 维度 | 现状 | 转型后 |
|------|------|--------|
| 展示价值 | 无法展示 | 可作为全栈项目展示 |
| 技术深度 | 被领域掩盖 | 架构设计清晰可见 |
| 社区参与 | 封闭 | 可开源协作 |
| 职业助力 | 潜在风险 | 加分项 |
| 持续动力 | 受限 | 公开项目更有维护动力 |

## 转型方向

### 推荐方案：通用本地媒体库管理器

**产品名称**: **MediaBox**（暂定，可替换）

**定位**: 面向本地视频收藏爱好者的现代化媒体库管理工具

**目标场景**:
- 电影收藏管理（个人本地影片库）
- 课程视频整理
- 家庭录像归档
- 创作者素材管理
- 任意本地视频文件的元数据管理

### 功能映射

```
当前功能                    转型后功能
─────────────────────────────────────────────────
影片库 (Movies)      →    媒体库 (Media Library)
番号解析             →    智能文件名解析
演员管理             →    创作者/演员管理
影片刮削             →    元数据自动获取
标签系统             →    智能标签分类
播放历史             →    观看追踪
收藏功能             →    收藏夹系统
精选帧               →    封面/缩略图管理
笔记评论             →    个人备注
```

### 差异化定位

相比现有方案（Plex、Jellyfin、Emby）的优势：

| 特性 | MediaBox | Plex/Jellyfin | 备注 |
|------|----------|---------------|------|
| 部署复杂度 | 零配置，开箱即用 | 需要服务器配置 | 个人用户友好 |
| 网络依赖 | 完全离线可用 | 部分功能需联网 | 隐私优先 |
| 资源占用 | 轻量级 | 较重 | 老旧设备可用 |
| 刮削灵活性 | 多源可配置 | 固定来源 | 容错性更强 |
| 定制化 | 开源可修改 | 受限 | 开发者友好 |

### 需要修改的组件

#### 1. 元数据刮削器（重构重点）

**当前**: 专注于特定成人元数据源

**转型后**: 支持通用视频元数据源

```go
// 新架构：插件化刮削器
interface MetadataProvider {
    Name() string
    Search(query string) ([]SearchResult, error)
    GetDetails(id string) (*MediaDetails, error)
    SupportsType(mediaType MediaType) bool
}

// 内置提供方
- TMDBProvider      // The Movie Database
- TVDBProvider      // The TVDB
- OMDBProvider      // Open Movie Database
- FileNameProvider  // 基于文件名的智能解析
- CustomProvider    // 用户自定义来源
```

#### 2. 数据模型调整

```go
// 通用媒体模型（替代特定模型）
type Media struct {
    ID          string
    Title       string
    OriginalTitle string
    Type        MediaType // Movie / TVShow / Video
    Year        int
    Duration    int
    Description string

    // 人员信息（替代演员）
    Cast        []Person  // 演员、讲者、创作者
    Directors   []Person
    Writers     []Person

    // 分类标签
    Genres      []string
    Tags        []string

    // 元数据
    Rating      float64
    Posters     []Image
    Backdrops   []Image

    // 文件信息
    Files       []MediaFile

    // 用户数据
    UserRating  int
    IsFavorite  bool
    Notes       string
    PlayedCount int
    LastPlayed  time.Time
}
```

#### 3. 文件名解析器

```go
// 支持多种命名约定
var namingPatterns = []Pattern{
    // 电影: Movie Name (2024).mp4
    {regex: `^(?P<title>.+?)\s*\((?P<year>\d{4})\)`, type: Movie},

    // 电视剧: Show Name - S01E02 - Episode Title.mp4
    {regex: `^(?P<title>.+?)\s*-?\s*[Ss](?P<season>\d+)[Ee](?P<episode>\d+)`, type: TVShow},

    // 课程: [Category] Course Name - Lesson 01.mp4
    {regex: `^\[(?P<category>.+?)\]\s*(?P<title>.+)`, type: Course},

    // 通用: Title - Date - Tag.mp4
    {regex: `^(?P<title>.+?)\s*-\s*(?P<date>\d{4}[-.]\d{2}[-.]\d{2})`, type: Video},
}
```

#### 4. 前端界面调整

```
界面映射:
影片详情页 → 媒体详情页（根据类型展示不同布局）
演员库     → 人员库（演员、导演、讲者）
标签浏览   → 分类浏览（类型 + 标签组合）
番号搜索   → 智能搜索（标题、人员、年份）
```

## 风险评估

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| **与原项目关联暴露** | 中 | 高 | 彻底清理提交历史；使用全新仓库；避免同时维护两个公开项目 |
| **转型工作量超预期** | 中 | 中 | 分阶段实施；保持原项目功能稳定；设定明确的里程碑 |
| **新领域竞争激烈** | 高 | 低 | 差异化定位（轻量、离线、简洁）；专注特定使用场景 |
| **维护动力不足** | 中 | 中 | 设定公开承诺；寻找共同维护者；建立用户反馈循环 |
| **开源后负面评价** | 低 | 中 | 完善文档和示例；积极回应 issue；保持代码质量 |

### 关键决策点

**决策 1**: 新仓库 vs 清理现有仓库
- **建议**: 新仓库更安全，避免提交历史泄露风险
- **代价**: 丢失 star/fork 历史（当前可能不多）

**决策 2**: 完全重写 vs 逐步迁移
- **建议**: 逐步迁移，保留核心架构
- **优势**: 工作量可控，保持功能稳定

**决策 3**: 单人维护 vs 寻找合作者
- **建议**: 初期单人维护，稳定后寻找合作者
- **理由**: 保持方向一致，避免早期分歧

### 术语对照表

| 当前术语 | 转型后术语 | 说明 |
|----------|------------|------|
| 影片 (Movie) | 媒体 (Media) | 通用化 |
| 演员 (Actor) | 人员 (Person) | 包含演员、导演等 |
| 番号 (ID) | 标识符 / 文件名 | 通用化 |
| 刮削 (Scrape) | 元数据获取 | 更易理解 |
| 库 (Library) | 库 / 收藏 (Collection) | 保持不变 |

# 双库维护架构设计方案

**日期**: 2026-03-26
**版本**: v1.0
**状态**: 规划阶段

---

## 问题分析

### 当前困境

你想同时维护两个版本：
- **版本 A** (Curated): 当前项目，功能完善但内容敏感
- **版本 B** (MediaBox): 转型后的通用版本，可公开展示

### 直接维护两个独立仓库的问题

| 问题 | 影响 | 示例 |
|------|------|------|
| 代码重复 | 任何改进需要手动同步两次 | 修复 bug 需要提两个 PR |
| 版本漂移 | 两个仓库逐渐不一致 | A 优化了扫描器，B 没有 |
| 测试负担 | 测试用例需要维护两份 | 测试工作量翻倍 |
| 认知负担 | 开发者需要记住两套代码 | 容易混淆实现细节 |
| 发布复杂 | 版本发布需要两次 | 容易遗漏或版本号不一致 |

---

## 核心思路：单仓库多构建目标

### 架构原则

**不要维护两个仓库，而是维护一个仓库，产出两个产品**

```
┌─────────────────────────────────────────────────────────────────┐
│                     单一源代码仓库                                │
│                     (Single Source of Truth)                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────┐      │
│   │              共享核心层 (90% 代码)                   │      │
│   │  • 文件系统扫描与管理                                │      │
│   │  • 数据库设计与存储                                  │      │
│   │  • HTTP API 框架                                     │      │
│   │  • 前端基础组件                                      │      │
│   │  • 播放历史与进度追踪                                │      │
│   │  • 通用工具函数                                      │      │
│   └─────────────────────────────────────────────────────┘      │
│                              │                                  │
│           ┌──────────────────┼──────────────────┐              │
│           ▼                  ▼                  ▼              │
│   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐      │
│   │  标准版构建   │   │  扩展版构建   │   │  开发版构建   │      │
│   │  (MediaBox)  │   │  (Curated)   │   │  (Dev)       │      │
│   └──────────────┘   └──────────────┘   └──────────────┘      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 核心优势

| 优势 | 说明 |
|------|------|
| **单一事实源** | 所有改进在一个地方进行，自动同步到所有版本 |
| **自动化构建** | CI/CD 自动产出多个版本，无需手动操作 |
| **共享测试** | 一套测试覆盖所有版本，确保质量一致 |
| **灵活扩展** | 未来可以轻松添加第三个版本（如 Pro 版） |
| **代码复用** | 90% 代码共享，只需维护差异部分 |

### 3. 开发工作流

#### 3.1 本地开发切换

```bash
# 切换到标准版开发
npm run dev:standard

# 切换到扩展版开发
npm run dev:extended
```

对应的 `package.json` 脚本：
```json
{
    "scripts": {
        "dev:standard": "scripts/switch-edition.js standard && vite --config build/configs/standard/vite.config.ts",
        "dev:extended": "scripts/switch-edition.js extended && vite --config build/configs/extended/vite.config.ts",
        "build:standard": "scripts/build-standard.sh",
        "build:extended": "scripts/build-extended.sh",
        "build:all": "npm run build:standard && npm run build:extended"
    }
}
```

#### 3.2 版本切换脚本

```javascript
// scripts/switch-edition.js
const fs = require('fs');
const path = require('path');

const edition = process.argv[2]; // 'standard' or 'extended'

// 创建符号链接或复制配置
const configSource = path.join(__dirname, '../editions', edition, 'frontend/src/config');
const configTarget = path.join(__dirname, '../core/frontend/src/edition-config');

// 清理旧的
if (fs.existsSync(configTarget)) {
    fs.rmSync(configTarget, { recursive: true });
}

// 创建新的链接
fs.symlinkSync(configSource, configTarget, 'junction');

console.log(`Switched to ${edition} edition`);
```

### 4. CI/CD 自动化

```yaml
# .github/workflows/build-all.yml
name: Build All Editions

on:
  push:
    branches: [master]
  pull_request:
    branches: [master]

jobs:
  # ============ 测试共享核心 ============
  test-core:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Test Core Backend
        working-directory: ./core/backend
        run: go test ./...

      - name: Test Core Frontend
        working-directory: ./core/frontend
        run: pnpm test

  # ============ 构建标准版 ============
  build-standard:
    needs: test-core
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
    steps:
      - uses: actions/checkout@v4

      - name: Build Standard Edition
        run: ./build/scripts/build-standard.sh

      - name: Upload Artifact
        uses: actions/upload-artifact@v4
        with:
          name: mediabox-${{ matrix.os }}
          path: dist/standard/

  # ============ 构建扩展版 ============
  build-extended:
    needs: test-core
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
    steps:
      - uses: actions/checkout@v4

      - name: Build Extended Edition
        run: ./build/scripts/build-extended.sh

      - name: Upload Artifact
        uses: actions/upload-artifact@v4
        with:
          name: curated-${{ matrix.os }}
          path: dist/extended/

  # ============ 发布标准版（开源） ============
  release-standard:
    needs: build-standard
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/master'
    steps:
      - uses: actions/download-artifact@v4
        with:
          pattern: mediabox-*
          merge-multiple: true

      - name: Create Release
        uses: softprops/action-gh-release@v1
        with:
          files: dist/standard/*
          tag_name: mediabox-v${{ github.run_number }}
          repository: yourname/mediabox  # 发布到公开仓库

  # ============ 发布扩展版（私有） ============
  release-extended:
    needs: build-extended
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/master'
    steps:
      - uses: actions/download-artifact@v4
        with:
          pattern: curated-*
          merge-multiple: true

      - name: Create Private Release
        uses: softprops/action-gh-release@v1
        with:
          files: dist/extended/*
          tag_name: curated-v${{ github.run_number }}
          repository: yourname/curated  # 发布到私有仓库
```

## 维护工作流

### 日常开发流程

```
1. 功能开发
   │
   ├── 如果是通用功能 → 修改 core/ 目录
   │                    └── 自动适用于两个版本
   │
   └── 如果是版本特定功能 → 修改 editions/<edition>/
                          └── 只影响指定版本

2. 测试
   │
   ├── 本地测试两个版本
   │   ├── npm run dev:standard
   │   └── npm run dev:extended
   │
   └── 确保修改没有破坏另一版本

3. 提交
   │
   └── 常规 git 提交流程

4. CI/CD 自动处理
   │
   ├── 自动测试两个版本
   ├── 自动构建两个版本
   └── 自动发布到各自仓库
```

### 冲突避免策略

| 场景 | 解决方案 |
|------|----------|
| 两个版本需要不同的业务逻辑 | 使用接口 + 版本特定实现 |
| 数据库 Schema 差异 | 核心表相同，差异字段用 JSON 列存储 |
| API 端点差异 | 核心路由相同，版本特定路由在各自目录注册 |
| 前端页面差异 | 共享布局组件，页面内容组件版本特定 |

### 渐进式重构路径

如果你当前只有一个版本（Curated），可以按以下步骤迁移：

```
阶段 1: 重构准备（2-3周）
├── 创建 core/ 目录结构
├── 将通用代码逐步迁移到 core/
├── 保留原有功能不变
└── 验证核心代码可独立运行

阶段 2: 引入多版本（2-3周）
├── 创建 editions/extended/ 目录
├── 将当前特定代码迁移到 extended/
├── 建立构建脚本
└── 验证可以成功构建原版本

阶段 3: 开发标准版（4-6周）
├── 创建 editions/standard/ 目录
├── 实现标准版刮削器
├── 调整前端界面
└── 验证标准版功能完整

阶段 4: 自动化与发布（2周）
├── 配置 CI/CD 多版本构建
├── 设置文档自动同步
├── 准备开源材料
└── 正式发布标准版
```

## 风险与应对

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| 架构复杂度过高 | 中 | 中 | 保持简单，避免过度设计；逐步演进 |
| 版本间代码耦合 | 中 | 中 | 明确接口边界；定期代码审查 |
| 开发环境配置复杂 | 低 | 低 | 提供一键切换脚本；完善文档 |
| 构建时间增加 | 中 | 低 | CI 并行构建；本地只构建开发版本 |
| 扩展版代码泄露 | 低 | 高 | 独立的私有仓库；CI 自动分离发布 |

## 推荐实施策略

### 我的建议

**采用"单仓库多构建目标"方案**，理由：

1. **维护成本低**: 90% 代码共享，bug 修复一次搞定
2. **自动化程度高**: CI/CD 自动产出两个版本，无需手动同步
3. **灵活性好**: 未来可轻松添加更多版本（如 Pro 版、Lite 版）
4. **技术可行**: Build Tags、Vite 别名等机制成熟可靠
5. **风险可控**: 代码在一起但发布分离，不会意外泄露

### 立即可以做的步骤

```bash
# 1. 在当前仓库创建新分支
git checkout -b refactor/multi-edition

# 2. 创建目录结构
mkdir -p core/backend core/frontend editions/standard editions/extended build/scripts

# 3. 开始迁移通用代码
# 逐步将 backend/internal 中通用部分移动到 core/backend/

# 4. 验证构建
# 确保现有功能不受影响
```

### 决策检查点

在开始之前，请确认：

- [ ] 你愿意投入时间重构现有代码吗？（预计 6-10 周）
- [ ] 当前项目功能已经稳定了吗？（建议在稳定期重构）
- [ ] 你有精力同时维护两个版本的后续迭代吗？

如果答案是肯定的，这个方案将为你提供一个可持续的、自动化的双版本维护方案。

# 代码质量审查与前后端改进建议

## 1. 审查范围

- 审查对象为当前仓库中可验证的代码与文档。
- 前端审查重点覆盖 `src` 下的路由、页面、组件、typed mock 数据与工程脚本。
- 后端审查重点覆盖当前是否存在真实后端实现，以及现有后端规划文档是否清晰区分“当前状态”和“目标设计”。
- `docs/film-scanner/` 下的 Go 工程与参考 SDK 不视为当前产品后端实现，只作为参考资料看待。

## 2. 总体结论

当前仓库的前端原型完成度较高，页面结构、浏览路径和产品信息架构已经比较清晰，但代码层面仍停留在“typed mock 驱动的高保真原型”阶段。现阶段的主要问题不是样式或组件堆砌，而是领域边界、状态一致性、错误兜底、测试体系和服务抽象尚未成型。

后端方面，当前仓库根目录并没有与前端形成真实对接关系的后端实现。后端更多停留在架构文档、约束文档和参考代码阶段，因此建议重点放在契约先行、模块边界、任务模型、错误模型和工程结构的前置设计上，而不是过早把参考 CLI 或外部 SDK 直接当作正式后端。

## 3. 主要审查发现

### 3.1 高优先级问题

#### 发现 1：无效影片 ID 会静默回退到第一条数据

`getMovieById()` 在找不到目标影片时，直接回退到 `movies[0]`，这会让详情页、播放器页和壳层顶部导航在参数非法时展示错误内容，而不是进入空态、404 或错误态。

影响：

- 用户会看到“错误但看似正常”的页面。
- 后续接入真实服务时，错误语义会被进一步放大。
- 前端很难区分“数据不存在”和“兜底展示”。

建议：

- 将 `getMovieById()` 改为返回 `Movie | undefined`。
- 在路由层补充 `NotFound` 或显式空态。
- 让 `detail`、`player`、壳层返回逻辑统一处理非法 ID。

#### 发现 2：收藏状态不是单一数据源，存在 UI 与数据脱节风险

`MovieCard.vue` 中的 `isFavorite` 使用本地 `ref(props.movie.isFavorite)` 保存，切换收藏时只更新局部状态，不回写上层数据，也没有在 `props.movie` 变化时重新同步。

影响：

- 卡片状态、列表过滤结果、详情页状态可能不一致。
- 后续接入服务层或全局状态后，这种局部副本会造成更多同步问题。

建议：

- 收藏状态改为上层统一管理。
- 卡片只负责触发 `toggleFavorite` 事件。
- 组件内部如需展示状态，应基于 `props` 或 `computed`，而不是脱离数据源的局部副本。

### 3.2 中优先级问题

#### 发现 3：浏览路由依赖 query 状态，但侧边栏切换未保留上下文

当前 `library / favorites / recent / tags` 共用同一浏览模型，搜索词 `q`、标签页 `tab` 和选中影片 `selected` 都通过 URL query 表达。但 `AppSidebar.vue` 中的导航只传递 `name`，没有根据产品策略保留或重置 query。

影响：

- 浏览体验不连续。
- 用户在不同浏览模式间切换时，搜索和选中态容易丢失。
- 与“连续浏览体验”的产品目标不一致。

建议：

- 明确哪些 query 应保留，哪些应重置。
- 统一封装浏览导航函数，而不是在多个组件内零散处理。

#### 发现 4：路由缺少 404、无效参数和异常加载兜底

当前路由仅定义了业务页面，没有补充 catch-all 路由，也没有对异步页面加载失败、非法参数和空数据做统一兜底。

影响：

- 出错时很难进入可预期的恢复路径。
- 后续接入真实后端后，错误体验会明显变差。

建议：

- 增加通配 404 路由。
- 统一非法参数处理策略。
- 为异步页面加载和服务异常预留错误态。

#### 发现 5：领域类型、mock 数据、查询逻辑、格式化方法耦合在同一模块

`src/lib/jav-library.ts` 同时承载领域模型、mock 数据生成、业务查询和格式化工具，当前开发成本低，但会显著增加未来替换服务层、添加适配器和编写测试的难度。

影响：

- 模块职责不清。
- 无法平滑引入 repository 或 service adapter。
- 纯函数测试边界不清晰。

建议：

- 拆分为 `types`、`mocks`、`formatters`、`repository` 或 `services`。
- 页面层依赖接口，不直接依赖 mock 常量。

#### 发现 6：虚拟瀑布流参数偏硬编码，扩展后有滚动测量风险

`VirtualMovieMasonry.vue` 中 `min-item-size` 与 `min-size` 固定为 `2200`，当前依赖块级估算高度工作。随着卡片内容、高度策略或响应式布局变化，这类硬编码值容易导致测量不准、滚动跳动或白屏区间。

建议：

- 重新校准测量策略。
- 尽量减少过于保守的固定高度估算。
- 为大数据量场景补充实际滚动验证。

#### 发现 7：缺少测试与静态检查门禁

当前 `package.json` 只有 `dev`、`build`、`preview`，没有 `lint`、`test`、`typecheck` 的显式质量门禁，也未看到前端测试文件。

影响：

- 回归风险高。
- 重构缺少验证手段。
- 模板层和交互层的问题只能依赖人工检查。

建议：

- 增加 `Vitest` + `Vue Test Utils`。
- 增加 `ESLint`，至少覆盖 TypeScript 与 Vue SFC。
- 在后续 CI 中执行 `lint + test + build`。

### 3.3 低优先级问题

#### 发现 8：仓库中仍存在脚手架残留文件与工程元数据噪音

- `src/components/HelloWorld.vue` 未被使用。
- `package.json` 的 `name` 仍为仓库标识 `jav-shadcn`；正式产品名为 **Curated**（见 `description` 与 `README`）。

建议：

- 删除无用脚手架文件。
- 统一工程命名，减少协作噪音。

## 4. 前端改进建议

### 4.1 近期优先落地

1. 建立服务层雏形：
   `views` 与 `components` 不再直接依赖 `jav-library.ts` 中的 mock 常量，而是依赖 `library service` 接口。
2. 修正实体获取语义：
   不再对非法影片 ID 静默兜底，统一进入 404、空态或错误态。
3. 收拢交互状态来源：
   收藏、评分、用户标签等可变状态由上层统一维护，子组件只发事件。
4. 明确浏览上下文策略：
   对 `q`、`tab`、`selected` 等 URL 状态定义保留与重置规则。
5. 增加质量门禁：
   至少补齐 `lint`、`test`、`typecheck` 脚本。

### 4.2 中期结构优化

1. 将 `src/lib/jav-library.ts` 拆分为更清晰的领域模块。
2. 为 `library`、`detail`、`player`、`settings` 提炼服务契约与 DTO。
3. 提炼可复用 composable，例如浏览 query 状态同步、影片选择、导航策略。
4. 为虚拟滚动和搜索过滤补充基准验证，避免在真实数据接入后退化。
5. 在页面级组件中补充空态、错误态、加载态三类基础状态模型。

### 4.3 推荐的前端目标结构

```text
src/
  components/
  composables/
  services/
    contracts/
    adapters/
      mock/
      desktop/
  lib/
    formatters/
    utils/
  domain/
    movie/
    library/
    player/
```

## 5. 后端改进建议

### 5.1 当前判断

- 当前仓库没有根级后端实现。
- 当前前端没有真实 API、桌面桥接或后端通信链路。
- 后端更多体现为设计文档和参考资料，而不是可运行交付物。

这意味着后端建议的重点应放在“如何正确落地”，而不是“如何修补现有业务代码”。

### 5.2 后端落地优先级建议

1. 先定义契约，再写服务：
   优先定义命令、事件、DTO、错误码、任务状态机。
2. 独立正式后端工程边界：
   使用明确的 `backend/` 目录或独立仓库，不要把 `docs/film-scanner/` 直接演化成正式产品后端。
3. 统一任务模型：
   扫描、搜刮、缓存、缩略图生成、播放器控制都应进入同一任务系统，而不是各自写同步脚本。
4. 错误模型稳定化：
   统一 `errorCode`、重试标记、用户可展示消息和诊断字段。
5. 第三方数据做映射隔离：
   `metatube-sdk-go` 或其他数据源的返回结构不能直接暴露给前端契约层。
6. 先做仓储和迁移边界：
   数据库迁移、repository、domain service 分层先定好，再往上叠加业务能力。

### 5.3 推荐的后端目标结构

```text
backend/
  cmd/
    curated/
  internal/
    contracts/
    app/
    domain/
    repository/
    task/
    player/
    scanner/
    scraper/
    config/
    logging/
  migrations/
```

### 5.4 后端与前端的对接建议

1. 前端只依赖统一服务接口，不直接依赖底层协议。
2. 后端返回结构以列表页、详情页、播放器页、设置页的实际信息架构为导向。
3. 所有异步能力都应支持任务进度与事件推送。
4. 前后端需要共享一套稳定的领域术语，如 `Movie`、`ScanTask`、`ScrapeTask`、`PlayerState`、`AppError`。

## 6. 建议的执行顺序

### 6.1 前端第一阶段

1. 新增 `services/contracts` 与 `services/adapters/mock`。
2. 重构 `getMovieById()` 的错误语义。
3. 修正收藏状态的数据流。
4. 增加 `lint` 与最小测试集。

### 6.2 后端第一阶段

1. 定义命令协议、事件协议、DTO 与错误码。
2. 设计任务表和任务状态流转。
3. 搭建最小 `backend/` 工程骨架。
4. 先实现 `listMovies / getMovieDetail / toggleFavorite` 这类最基础能力。

## 7. 结论

这个仓库当前最大的优点是产品方向明确、前端原型完整、未来桌面化路径已经被文档化；最大的风险则是原型层直接承载了过多未来责任，导致“看起来像完整产品，实际上还停留在 mock 阶段”。

因此，最值得优先投入的不是继续堆页面，而是补齐前端服务边界、状态一致性、错误兜底和测试门禁，并让后端以契约驱动方式独立落地。这样后续无论接 `Electron`、`Go`、`SQLite` 还是 `mpv`，都不会反向破坏当前前端结构。

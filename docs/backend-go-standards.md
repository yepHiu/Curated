# Curated 后端 Go 实现规范

## 1. 目的

本文用于约束未来 `backend/` 下的 `Go` 工程组织方式，确保模块边界、错误处理、日志和测试风格保持一致。

## 2. 目录结构约束

推荐目录结构：

```text
backend/
  cmd/javd/
  internal/
    app/
    config/
    logging/
    contracts/
    library/
    scanner/
    scraper/
    assets/
    player/
    storage/
    tasks/
```

约束如下：

- `cmd/javd` 只负责启动和装配，不堆业务逻辑
- `internal/contracts` 负责命令、事件、DTO、错误码
- `internal/storage` 负责数据库访问与 migration
- 业务模块按领域分包，不创建巨型 `service` 或 `utils`
- 公共逻辑优先放入明确语义的包，而不是无边界公共目录

## 3. 分层约束

建议采用以下职责分层：

- `app`：装配与生命周期管理
- `service` 风格逻辑：领域业务编排
- `repository` 风格逻辑：数据访问
- `contracts`：对外契约
- `tasks/events`：异步能力

约束如下：

- 领域服务不直接依赖前端语义
- repository 不承担业务决策
- 控制层不拼装复杂业务数据
- 不跨模块直接读写彼此内部状态

## 4. 错误处理约束

- 不吞错，不写空 `catch` 风格逻辑
- 错误向上返回时应保留上下文
- 业务错误优先映射到稳定错误码
- 可重试错误与不可重试错误需要区分
- 不把第三方库原始错误文本直接暴露为最终用户错误

## 5. 日志约束

- 使用结构化日志
- 长耗时操作必须带任务或请求上下文
- 扫描、搜刮、缓存、播放器都要输出可追踪日志
- 日志应包含关键字段，如 `taskId`、`movieId`、`num`、`path`
- 不在日志中输出无意义大段文本或重复刷屏信息

## 6. 测试约束

优先补齐以下测试：

- 番号解析测试
- 扫描去重测试
- 仓储层集成测试
- DTO 映射测试
- 任务状态流转测试
- 播放器控制 mock 测试

测试要求：

- 纯逻辑优先单测
- 数据访问优先集成测试
- 外部依赖优先 mock 或适配器隔离
- 关键解析规则必须有覆盖样例

## 7. 第三方依赖约束

- `metatube-sdk-go` 必须经过 `scraper` 适配层调用
- `mpv` 必须经过 `player` 模块调用
- 数据库细节必须收敛在 `storage`
- 不允许在多个业务包中散落重复的第三方调用逻辑

## 8. 与 `film-scanner` 的关系

`docs/film-scanner` 可作为扫描与搜刮的参考样本，但在 `Go` 实现上应遵循以下规则：

- 复用其思路，不照搬其 CLI 结构
- 将其扫描、提号、搜刮、下载逻辑拆分到独立包
- 所有异步流程都纳入本项目任务系统
- 所有输出都对齐本项目 contracts，而不是直接复用 CLI 输出格式

# 侧边栏「继续播放」入口需求草案

Date: 2026-05-01
Status: Draft for review
Prototype: `docs/plan/2026-05-01-active-playback-sidebar-return-prototype.html`

## 背景

当前 Curated 的播放页是一个路由页：用户进入 `player/:id` 后，播放器组件负责解析播放地址、记录播放进度，并在离开播放页时把进度写入现有的播放进度存储。用户如果临时跳到首页、资料库、演员、设置等页面查看信息，页面会离开播放器。此时想回到刚才的影片，需要重新找影片，或进入观看历史再点一次继续播放。

这个体验问题的核心不是播放进度丢失，而是「正在看的内容没有一个稳定的回到入口」。现有观看历史能解决长期恢复，但不能解决同一次浏览会话内的快速返回。

## 用户需求整理

用户在播放页看影片时，如果跳转到其他页面，只是想临时查看资料或设置；离开后希望应用持续提醒「刚才有内容看到这里」，并提供一个明显但不打扰的入口，一点即可回到原播放页并从当前进度继续。

期望入口位置：左侧侧边栏底部区域，放在后端连接状态卡片的上方。这个位置接近全局状态区，用户在任何主页面都容易看到，同时不挤占主内容区。

## 目标

- 在用户离开播放页后，侧边栏显示一个「继续播放」入口。
- 入口点击后回到对应影片的播放页，并带上最近播放时间点。
- 入口只在有当前会话内的播放上下文时显示，不把普通观看历史都塞进侧边栏。
- 不引入后台播放或 mini player；本需求先解决「回到刚才播放页」。
- 视觉上保持 Curated 现有 AppShell / AppSidebar 的深色、紧凑、低干扰风格。

## 非目标

- 不让视频在用户离开播放页后继续后台播放。
- 不新增悬浮小窗、画中画托盘或全局音视频控制条。
- 不替代首页「继续观看」和观看历史页。
- 不处理跨浏览器刷新后的会话恢复；刷新后仍以现有观看历史/播放进度为主。
- 不把外部播放器真实状态同步回来。当前浏览器无法可靠知道 PotPlayer/mpv 等外部播放器是否还在播放。

## 文案判断

推荐主文案使用「继续播放」，而不是默认使用「正在播放」。

原因：当前 Web 播放器离开播放页后，`PlayerPage` 会卸载，实际视频不会继续在后台播放。如果侧边栏写「正在播放」，容易让用户误以为 Curated 在后台保持播放。更准确的表达是：

- 主标题：`继续播放`
- 状态行：`刚才播放至 42:16`
- 按钮/点击 aria：`回到播放页继续播放《影片标题》`

如果未来实现真正的 mini player 或后台音视频保持，再把状态文案升级为「正在播放」。

## 方案选项

### 方案 A：仅依赖现有观看历史

做法：离开播放页后不维护新的会话状态，只在侧边栏展示最近一条未完成观看历史。

优点：

- 实现最少。
- 刷新后也能显示。

缺点：

- 容易把长期历史误当作当前正在看的内容。
- 用户可能只是昨天看过一部片，今天打开应用也被侧边栏提示，噪音较高。
- 很难准确表达「刚才那个播放器」。

不推荐作为本需求的首选。

### 方案 B：当前 SPA 会话内的主动播放状态

做法：新增一个小型 module-scoped composable，例如 `useActivePlaybackSession()`。`PlayerPage` 在播放源加载、播放/暂停、时间更新、离开页面前，把当前影片、进度、状态、返回播放页的 route target 写入该 composable。`AppSidebar` 消费它，在非播放器路由显示「继续播放」卡片。

优点：

- 精准对应「我刚才离开播放器」这个场景。
- 不污染长期历史。
- 不需要引入 Pinia 或大型全局状态库，符合当前项目 Composables + 服务层模式。
- 能保留原播放器 route query，例如 `back=`、`browse=`、`t=`，点击返回时语义稳定。

缺点：

- 刷新后状态消失。
- 需要在 `PlayerPage` 和 `AppSidebar` 之间增加一个新的共享状态边界。

推荐作为首版实现。

### 方案 C：全局 mini player / 后台播放

做法：播放状态和 video 元素提升到 AppShell 或全局播放器层，路由切换时视频仍持续播放，侧边栏入口变成真正的正在播放控制。

优点：

- 最接近「正在播放」的字面含义。
- 可扩展成小窗、全局控制、后台播放。

缺点：

- 架构改动大，会影响播放器生命周期、HLS session 清理、键盘快捷键、沉浸式 HUD、路由返回、移动端布局。
- 当前需求只是快速回到播放页，不需要一次性承担这么大范围。

不建议本轮做。

## 推荐方案

采用方案 B：当前 SPA 会话内的「继续播放」入口。

### 状态模型

建议状态字段：

- `movieId`
- `title`
- `posterUrl` 或现有封面解析结果，可选
- `positionSec`
- `durationSec`
- `status`: `playing | paused | waiting | ended | error`
- `updatedAt`
- `routeTarget`: 回到播放器的 `RouteLocationRaw`

显示条件：

- 有 `movieId`。
- 当前路由不是同一影片的 `player` 页。
- 状态不是 `ended`。
- 进度没有接近结尾；可沿用现有 `getResumeSecondsForOpenPlayer` 的思路，低于 5 秒或超过 95% 时不显示。

清理条件：

- 播放结束或进度达到接近结尾。
- 当前影片从资料库不可用时，点击后由播放器现有 not-found 状态承接；后续实现可在侧栏直接隐藏。
- 用户手动点击卡片上的关闭按钮时，本轮会话内隐藏该条，直到新的播放状态产生。

### 数据流

1. `PlayerPage` 成功拿到 `playbackSrc` 后初始化 active playback state。
2. `@play` / `@pause` / `waiting` / `timeupdate` 或现有进度刷新点更新状态。
3. 离开播放页前刷新一次 `positionSec`，保证侧栏显示的是最新时间点。
4. `AppSidebar` 根据 active playback state + 当前 route 决定是否显示入口。
5. 用户点击入口后进入 `player/:id?t=<position>&autoplay=1`。

路由策略建议：优先复用当时播放器自己的 route，并只更新 `t` 和 `autoplay=1`。这样用户从资料库、历史、萃取帧等来源进入播放器时，原有 `back=` 语义不会被侧边栏入口破坏。

### UI 设计

展开侧边栏：

- 在后端状态卡上方显示一个小卡片。
- 主标题为 `继续播放`。
- 第二行显示影片标题，最多两行或单行截断。
- 第三行显示 `刚才播放至 42:16 / 1:58:20`。
- 底部显示一条细进度条。
- 整卡可点击，右侧可有一个小的播放箭头图标；可选提供关闭按钮。

收起侧边栏：

- 在后端状态点上方显示一个 40x40 左右的 icon button。
- 图标使用播放符号或 clapper/play 组合，外围显示极细进度环或底部短进度线。
- hover/title 显示完整文案：`继续播放：影片标题，42:16`。

移动端侧边栏抽屉：

- 复用展开侧边栏卡片。
- 点击后关闭抽屉并跳转到播放器。

视觉原则：

- 使用现有语义 token：`bg-background`、`bg-sidebar`、`border-border`、`text-muted-foreground`、`text-primary`。
- 不使用新的强装饰渐变，不抢主导航层级。
- 卡片要比后端状态卡更可点击，但不能像主 CTA 一样突兀。

## 可访问性

- 展开卡片使用真实按钮或 `RouterLink`。
- `aria-label` 包含影片标题和时间点，例如：`回到播放页继续播放《...》，当前 42 分 16 秒`。
- 关闭按钮如果存在，必须有独立 `aria-label`，避免误点。
- 收起态 icon button 必须有 `title` 和 `aria-label`。
- 进度条仅做辅助展示时使用 `aria-hidden="true"`，不要让屏幕阅读器重复朗读。

## 测试建议

首轮实现后建议覆盖：

- `useActivePlaybackSession`：保存、更新、结束清理、手动隐藏。
- `AppSidebar`：有 active playback 且不在 player 时显示；在同一 player 路由时隐藏；compact 模式显示 icon 入口。
- `PlayerPage`：播放源加载后写入状态；时间更新/离开页面前更新 `positionSec`。
- 路由点击：入口生成的 player route 带 `autoplay=1` 和最新 `t`，并保留原 `back=` 语义。

## 验收标准

- 用户从播放页跳到其他主页面后，侧边栏后端状态上方出现「继续播放」入口。
- 点击入口能回到刚才影片的播放页，并尽量从最近时间点继续。
- 当前就在播放器页面时，不显示重复入口。
- 影片播放结束、接近结尾或用户关闭提示后，入口不再显示。
- 展开、收起、移动端抽屉三种侧边栏形态都不出现文字溢出或布局挤压。
- Mock 模式和 Web API 模式都可工作；Mock 只使用当前前端会话状态，不要求写后端。

## 待确认点

1. 主文案是否采用推荐的「继续播放」，而不是「正在播放」。
2. 卡片是否需要关闭按钮；如果不需要，入口会一直显示到播放结束、切换影片或刷新应用。
3. 点击入口后是否总是 `autoplay=1`。我建议是，因为这是用户主动点击，符合“回去继续播放”的预期。
4. 收起侧边栏是否接受 icon-only 入口加 tooltip，不显示影片标题。

# 预览图比例适配需求与实现方案

## 1. 背景

当前影片详情页的预览图区域，使用的是固定 `16:9` 的网格卡片：

- 入口文件：`src/components/jav-library/DetailPage.vue`
- 当前实现：`CardContent` 使用固定列数的 `grid`
- 每个预览图按钮使用固定 `aspect-[16/9]`
- 图像组件 `MediaStill.vue` 默认以 `layout="fill"` + `fit="cover"` 渲染

这意味着：

- 横图、竖图、接近方图，都会被放进同一种固定比例容器
- 竖图和特殊宽图在预览区的展示不够自然
- 用户看到的是“统一的盒子”，不是“按图片自身比例展示的预览图”

用户本次提出的目标很明确：

- 预览图展示时，不要再强行统一为固定 `16:9`
- 预览图应根据图片自身比例自动适配
- 整体上仍然保持一排内视觉高度一致，避免布局散掉

## 2. 当前代码现状

### 2.1 已确认的现状

详情页预览图库当前逻辑在 `DetailPage.vue`：

- `previewImages` 的数据结构只是 `string[]`
- 前端目前只拿到了图片 URL，没有宽高或比例元数据
- 每张图都是：
  - 固定比例按钮容器：`aspect-[16/9]`
  - 内部 `MediaStill` 充满容器

相关限制是：

1. 前端在首次渲染前，并不知道某张预览图到底是横图还是竖图
2. 如果不补比例信息，就无法在初始布局阶段直接给出准确宽度
3. 目前的 UI 结构更像“固定海报格子”，而不是“自适应比例的预览图库”

### 2.2 当前问题的本质

这不是单纯的 `object-fit` 问题，而是：

- **容器比例被写死了**
- 而图片真实比例没有参与布局计算

所以只把 `object-cover` 改成 `object-contain` 并不能真正满足需求。那样只会变成：

- 外框还是固定 `16:9`
- 图片虽然不裁切，但会在固定盒子里留下大量空白
- 最终仍然不是“按比例适配展示”

## 3. 需求文档

### 3.1 目标

在影片详情页的预览图区域，实现“同一行高度统一、宽度按图片真实比例自适应”的展示效果。

### 3.2 需求范围

本次需求优先覆盖：

- `src/components/jav-library/DetailPage.vue` 中的预览图库

本次不强制覆盖：

- 大图查看器主图区域（当前已是 `object-contain`，问题不大）
- 大图查看器下方缩略图条
- 列表页海报卡片
- 首页推荐位海报

### 3.3 功能要求

1. 预览图应按图片实际宽高比展示
2. 同一排预览图保持统一高度
3. 图片不应再被固定 `16:9` 容器强制裁切
4. 预览图仍然可点击，点击后继续打开现有预览查看器
5. 布局应支持横图和竖图混排
6. 极端宽图或极端高图不能把整排布局破坏掉

### 3.4 视觉与交互要求

1. 每张预览图保持统一卡片高度
2. 每张卡片宽度由图片比例决定
3. 卡片保留现有圆角、边框、hover、focus 样式基调
4. 图片未加载完成前，预览区不能明显塌陷
5. 图片加载失败时，卡片仍保留稳定占位

### 3.5 非目标

本次不做以下内容：

- 不重做大图查看器交互
- 不引入全新的图库组件库
- 不一次性改造所有图片展示场景
- 不要求后端在本期必须返回图片尺寸元数据

## 4. 候选方案

### 方案 A：前端本地测量图片比例后自适应展示

做法：

- 详情页在渲染预览图时，前端通过图片加载事件读取 `naturalWidth / naturalHeight`
- 在组件本地维护每张图的 `aspectRatio`
- 预览区从固定 `grid` 改成 `flex-wrap` 或“按固定高度排版”的图库布局
- 每个卡片统一高度，宽度根据 `aspectRatio` 计算

优点：

- 不改后端 API
- 改动范围集中在前端详情页
- 可以较快落地

缺点：

- 首次加载前拿不到比例，初次布局需要 fallback
- 图片加载后会有轻微重排风险

### 方案 B：后端返回预览图尺寸元数据

做法：

- 后端在详情接口中返回 `previewImages` 的同时，附带每张图的 `width` / `height` 或 `aspectRatio`
- 前端初次渲染时即可直接按真实比例布局

优点：

- 首屏布局最稳定
- 减少前端二次测量和重排
- 更适合后续做虚拟化、缓存和统一图片能力建设

缺点：

- 需要改后端 DTO、抓取/存储链路、前端类型和映射
- 本期成本更高

### 方案 C：保留固定网格，仅改成 contain

做法：

- 仍保留 `aspect-[16/9]`
- 仅把图片改成 `object-contain`

优点：

- 改动最小

缺点：

- 不能满足“按比例适配展示”的核心需求
- 视觉上仍然是固定盒子，不是自适应图库

**结论：不推荐。**

## 5. 推荐方案

推荐采用：

- **短期实施：方案 A**
- **中期演进：视效果再决定是否升级到方案 B**

原因：

1. 用户当前要解决的是展示体验问题，优先级在前端可见效果
2. 当前 `previewImages` 只是 URL 列表，前端先本地测量能最快落地
3. 如果后续确认需要“零重排、零猜测”的初始布局，再补后端比例元数据更合适

## 6. 推荐实现方案细化

### 6.1 数据结构

当前：

- `previewImages: string[]`

推荐在详情页本地增加一个派生结构，不先改 API：

- `previewGalleryItems: Array<{ url: string; aspectRatio?: number; status: "idle" | "loaded" | "error" }>`

说明：

- `url`：原始图片地址
- `aspectRatio`：前端测量后的 `width / height`
- `status`：用于 skeleton、失败占位和调试

### 6.2 布局结构

把当前固定列数 `grid` 改成更适合比例自适应的布局：

- 父容器改为 `flex flex-wrap gap-3 items-start`
- 每个预览图卡片使用固定高度，例如：
  - 移动端：`h-36`
  - 桌面端：`h-40` 或 `h-44`
- 卡片宽度由比例决定：
  - `width = height * aspectRatio`
  - 在 Vue 中通过 `style` 绑定 `aspect-ratio`

推荐卡片规则：

1. 固定高度
2. 宽度按比例展开
3. 设置最小宽度和最大宽度，防止极端比例破坏排版

示意约束：

- `min-width: 8rem`
- `max-width: min(100%, 24rem)` 或按断点控制

### 6.3 图片渲染策略

卡片容器比例与图片比例一致后，图片本身建议：

- 使用完整展示优先
- `MediaStill` 增加或复用适合该场景的模式
- 预览图内部优先使用 `fit="contain"` 或与容器同比分布的非裁切模式

目标不是铺满裁切，而是：

- **尽量完整显示图片本体**
- 让卡片尺寸承担比例适配，图片不再被强裁

### 6.4 比例获取方式

推荐前端本地测量：

1. 详情页拿到 `previewImages`
2. 为每个 URL 创建轻量测量流程
3. 在图片 `load` 后读取：
   - `naturalWidth`
   - `naturalHeight`
4. 写入本地 `aspectRatio`

比例回退规则：

- 未加载前默认按 `16 / 9`
- 加载失败时保留默认比例或使用保守比例

这样可以保证：

- 即使还没拿到真实比例，也有稳定占位
- 加载完成后再切换为真实比例

### 6.5 极端比例保护

必须增加边界保护，否则超宽或超高图片会破坏布局。

推荐策略：

1. 不裁切图片主体
2. 允许比例自适应
3. 但对展示宽度设置上下限

建议规则：

- 非常高的图：保证不窄到无法点击
- 非常宽的图：保证不宽到独占整行或挤坏布局

### 6.6 Skeleton 与加载稳定性

当前预览区不能在图片尚未加载时塌陷。

建议：

1. 每张卡片先渲染固定高度占位
2. 宽度先使用默认比例
3. 图片完成测量后再平滑更新为真实比例

如果后续发现重排仍明显，再考虑升级到后端返回比例元数据。

## 7. 实现计划

### 阶段 1：前端自适应比例落地

目标：

- 在不改后端的前提下，让详情页预览图库支持按图片真实比例展示

涉及文件：

- `src/components/jav-library/DetailPage.vue`
- `src/components/jav-library/MediaStill.vue`
- 可能新增：
  - `src/composables/use-preview-image-ratios.ts`
  - 或 `src/lib/preview-image-ratios.ts`

实施步骤：

1. 梳理详情页预览图区当前 DOM 结构和点击逻辑
2. 为预览图增加本地比例状态管理
3. 将预览区从固定 `grid` 改为固定高度的自适应换行布局
4. 给单卡增加最小/最大宽度保护
5. 保留现有点击、hover、focus、打开查看器行为
6. 处理加载失败和未测量完成时的 fallback 展示

### 阶段 2：测试与回归

目标：

- 确保比例适配不影响现有详情页功能

建议补测内容：

1. 横图、竖图、近方图混排时，卡片高度一致
2. 点击任意预览图，仍正确打开查看器并定位到对应索引
3. 图片加载失败时，预览区不塌陷
4. 小屏与桌面断点下布局都不破

建议关注的现有测试文件：

- 详情页相关测试文件
- 预览查看器相关测试文件

### 阶段 3：可选增强

如果阶段 1 上线后仍存在以下问题：

- 首次进入详情页时卡片重排明显
- 弱网/慢盘环境下预览区抖动感明显
- 后续要把同类能力推广到更多图片场景

则进入可选增强：

- 扩展 API，让后端返回预览图尺寸元数据

涉及面将包括：

- `src/api/types.ts`
- `src/domain/movie/types.ts`
- `src/services/adapters/web/mappers.ts`
- 后端详情接口 DTO / 映射层

## 8. 验收标准

验收通过需满足：

1. 详情页预览图区不再固定为统一 `16:9`
2. 横图与竖图能按各自比例展示
3. 同一排视觉高度保持一致
4. 点击预览图仍能打开当前查看器
5. 不出现明显布局塌陷、超宽撑爆、超窄不可点的问题
6. 弱网或慢速存储场景下，预览区仍有稳定占位

## 9. 结论

这次需求的核心，不是简单调一个 `object-fit`，而是把“图片真实比例”纳入预览图库布局计算。

推荐落地路径是：

1. 先在前端详情页本地测量图片比例
2. 把预览区改为“固定高度 + 宽度按比例自适应”的布局
3. 为极端比例加边界保护
4. 若后续仍需更稳的首屏布局，再升级为后端返回比例元数据

这个方案能以较小改动，较快满足你现在的核心诉求。

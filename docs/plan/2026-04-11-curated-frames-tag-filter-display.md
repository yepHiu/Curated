# 2026-04-11 萃取帧标签筛选区设计

## 状态

- 状态：Approved
- 主题：在萃取帧库页面提供可见的标签展示与点击筛选入口
- 结论方向：顶部标签筛选区

## 背景

当前 Curated 已支持为萃取帧保存标签，并且在萃取帧详情弹窗中编辑标签，也已经存在“从标签跳回本页筛选”的行为。

缺口不在标签数据本身，而在主浏览界面里没有一个稳定、直接、可发现的“标签浏览入口”。用户无法一眼看到自己打过哪些标签，也无法自然地从标签进入对应的萃取帧集合。

## 目标

- 在萃取帧页面提供一个明显的标签展示区域
- 用户可以点击标签后筛选命中该标签的萃取帧
- 标签展示服务于“浏览和过滤”，而不是增加卡片信息密度
- 第一版保持低复杂度，不引入多选组合和复杂布尔逻辑

## 不做的事

- 第一版不把标签大量铺进每张萃取帧卡片主体
- 第一版不做多标签 AND / OR 组合筛选
- 第一版不做独立左侧标签导航栏
- 第一版不改动标签编辑模型，只补展示与筛选入口

## 推荐方案

### 1. 布局位置

在萃取帧页面中，将“标签筛选区”放在搜索 / 工具栏下方、萃取帧网格上方。

这样做的原因：

- 用户进入页面后，先看到可操作的筛选入口
- 标签的语义更接近“页面过滤器”，而不是“卡片元信息”
- 不会破坏现有卡片密度和视觉节奏

### 2. 展示形式

标签区域由以下元素组成：

- `全部` 入口
- 若干可点击标签 chip
- 每个标签右侧显示命中数量，例如 `构图 12`
- 当标签很多时，默认只展示前 12 到 16 个，剩余通过“更多标签”展开

排序建议：

- 第一优先级：标签命中数量降序
- 第二优先级：同数量时按最近使用或字母序稳定排序

这样可以让高频标签优先暴露，避免长尾标签把区域撑满。

### 3. 交互行为

第一版使用“单选标签筛选”：

- 默认状态为 `全部`
- 点击某个标签后，仅展示包含该标签的萃取帧
- 再次点击该标签可取消筛选，回到 `全部`
- 点击 `全部` 也可清空当前标签筛选

这样做的原因：

- 交互最直观
- 与用户当前诉求完全一致
- 不会引入组合筛选状态解释成本

### 4. 与现有搜索的关系

建议标签筛选区与现有页面搜索并存，且采用叠加过滤：

- 未输入搜索词时，只按标签筛选
- 输入搜索词后，结果为“当前搜索结果”与“当前标签筛选”的交集

这样符合用户对页面筛选的一般预期，也避免标签点击后跳转到完全不同的查询语义。

### 5. 卡片层展示策略

第一版不要求在每张萃取帧卡片内显式展示完整标签列表。

如需轻提示，只建议做弱化处理，例如：

- 在卡片 hover 或选中态显示当前命中的标签
- 或在详情面板中显示该帧标签

原则是：标签浏览入口集中在顶部，卡片继续承担“内容预览”角色，不把视觉重心打散。

## 边界情况

### 1. 无标签

当库中还没有任何萃取帧标签时，标签区显示轻提示：

`还没有萃取帧标签，给帧打标签后可在这里筛选。`

### 2. 标签很多

当标签数量较多时：

- 默认显示一部分高频标签
- 提供“更多标签”展开全部
- 不建议第一版做横向滚动条

### 3. 当前标签筛选结果为空

当选中某标签但没有命中结果时，保留当前筛选态，并显示空结果提示，例如：

`没有命中该标签的萃取帧。`

同时提供快速返回 `全部` 的入口。

## 实现边界建议

- 优先复用现有萃取帧标签数据来源，不重复造标签模型
- 优先在萃取帧页面内部完成筛选状态管理
- 筛选区与详情弹窗标签编辑保持语义一致，但不要把编辑逻辑塞进筛选区

## 后续可扩展项

如果第一版验证有效，后续可以继续扩展：

- 多标签筛选
- 标签搜索
- 常用标签 / 最近使用标签
- 卡片弱提示优化
- 与后端标签聚合接口对接，避免仅依赖已加载列表

## 当前建议

先做一个简单、清晰、可发现的顶部标签筛选区，把“看到自己打过的标签”和“点击后过滤萃取帧”这两个核心目标先完成。

这个方案最符合当前诉求，也最不容易把页面做乱。

## 补充：标签 / 影片 / 萃取帧数的展示与检索优化建议

### 1. 当前问题判断

如果目标不只是“能按标签点进去筛选”，而是让用户快速理解一个标签到底覆盖了多少内容，那么当前只显示：

- 标签名
- 萃取帧数

还不够。

原因是用户在浏览标签时，通常真正关心的是三个层次：

- 这个标签下有多少张萃取帧
- 这些萃取帧分布在多少部影片里
- 点进去以后，检索结果是否稳定、是否容易继续缩小范围

只显示帧数会有两个常见误判：

- 某标签帧数很多，但其实集中在一两部影片里，覆盖面并不广
- 某标签帧数不算高，但分布在很多影片里，反而更适合做入口标签

所以如果要优化“展示 + 检索”，推荐把标签视图从单一计数升级为“小型 facet 导航”。

### 2. 推荐方向

推荐把标签区域的统计信息升级为：

- 标签名
- `frameCount`：命中的萃取帧数
- `movieCount`：命中的去重影片数

展示形式建议保持轻量，例如：

- `构图 48 帧 · 12 片`
- `制服 17 帧 · 9 片`

这样做的好处是：

- 用户能区分“高频单片标签”和“跨片共性标签”
- 标签排序可以不再只按帧数，而是支持更有价值的排序策略
- 后续如果做演员维度、影片维度联动，这两个数字也能复用

### 3. 三种可选方案

#### 方案 A：只显示帧数，维持当前实现

优点：

- 成本最低
- 不改接口结构
- 适合先验证标签筛选入口是否被使用

缺点：

- 信息密度不足
- 容易误导用户对标签“覆盖范围”的判断
- 后续继续扩展时大概率还要重做 facet 模型

#### 方案 B：显示帧数 + 影片数，排序仍以帧数优先

优点：

- 是当前产品阶段最稳妥的升级
- 用户能快速理解标签价值
- 不会把标签区做成复杂分析面板

缺点：

- 仍然偏“热门标签优先”，长尾但跨片的重要标签可能不够显眼

#### 方案 C：显示帧数 + 影片数，并支持切换排序

例如支持：

- 按帧数排序
- 按影片数排序
- 按最近使用排序

优点：

- 信息最完整
- 适合萃取帧库逐步演进为真正的素材图库

缺点：

- 交互复杂度会上升
- 当前页面还没有必要立刻引入排序切换控件

### 4. 我的建议

我建议优先采用方案 B。

也就是：

- 标签区域展示 `标签名 + 帧数 + 影片数`
- 默认按 `frameCount DESC` 排序
- 同数量时按 `movieCount DESC`
- 再同数量时按标签名字稳定排序

这是当前最平衡的做法。

它比现状多提供一个关键判断维度，但不会把页面从“浏览页”推成“分析页”。

### 5. 检索交互建议

如果补上影片数统计，检索行为也建议一起收敛：

#### 5.1 标签筛选保持单选

当前仍建议保持：

- 单个激活标签
- 与搜索词做交集
- 再次点击取消

不要立刻做多标签 AND / OR。

原因很直接：

- 一旦引入多标签组合，用户会期待更强的结果解释
- `frameCount` / `movieCount` 也会变成“当前筛选上下文下的动态统计”
- 前后端复杂度会明显上升

#### 5.2 搜索框继续做全文入口，标签区做 facet 入口

建议职责明确分开：

- 搜索框：输入自由文本，负责全文检索
- 标签区：点击已知标签，负责结构化过滤

不要让标签区承担“搜索标签名”的责任，不然顶部会变成两个搜索入口，信息层级会乱。

#### 5.3 点击标签后，结果区最好同步反馈当前上下文

当前筛选条建议明确展示：

- 当前标签
- 命中帧数
- 命中影片数

例如：

- `当前标签：构图 · 48 帧 · 12 片`

这样用户在滚动很长列表时，不会忘记自己当前看到的是哪一层结果。

### 6. 接口与数据模型建议

如果决定优化这块，后端 facet 接口建议从当前的：

- `name`
- `count`

升级为：

- `name`
- `frameCount`
- `movieCount`

当前 `GET /api/curated-frames/tags` 已经有聚合入口，最合适的做法不是在前端临时二次扫全量结果，而是让后端直接返回完整 facet。

原因：

- Web API 模式下，萃取帧库已经走分页，不应该为了一个统计区再拉全量数据
- `movieCount` 是去重聚合，放在后端做更稳定
- 后续若做“只看当前 actor / 当前 movie 上下文下的标签 facet”，也更容易扩展

建议接口演进为：

- `GET /api/curated-frames/tags`
- 入参可选带当前上下文：`q`、`actor`、`movieId`
- 返回每个标签在当前上下文下的 `frameCount` 与 `movieCount`

这样标签区会真正成为“上下文感知的 facet 区”，而不是静态词云。

### 7. 展示层建议

UI 上不建议把这块做得太重。比较合适的结构是：

- 第一行：标题 `标签筛选`
- 第二行：标签 chip 列表
- 每个 chip 内显示 `标签名 + 简短统计`

不建议：

- 做成表格
- 做成左侧独立统计面板
- 同时塞入排序下拉和搜索框

因为当前萃取帧页的主角仍然应该是内容网格，不是统计区。

### 8. 推荐实施顺序

如果后续要真正优化，我建议按下面顺序推进：

1. 先把标签 chip 从“标签名 + 帧数”升级到“标签名 + 帧数 + 影片数”。
2. 再把当前筛选提示条补成“当前标签 + 命中帧数 + 命中影片数”。
3. 最后才考虑是否需要“按影片数排序”切换或“上下文动态 facet”。

这个顺序的好处是：

- 每一步都能独立交付
- 每一步都能直接提升可理解性
- 不会一次把检索模型改得太重

## 补充结论

如果你的重点是“展示标签、影片、萃取帧数，并让检索更有判断力”，那我不建议只停留在标签旁边显示一个 `count`。

更合理的最小升级是：

- 标签显示 `frameCount + movieCount`
- 搜索与标签继续做交集
- 结果区同步展示当前上下文统计

这会让用户更快判断一个标签值不值得点，也能让萃取帧页从“能筛选”提升到“筛选结果可理解”。

## 补充：搜索框功能优化建议

### 1. 当前搜索框的实际状态

如果这里指的是当前萃取帧页顶部搜索框，那么它现在更接近“单框全文 contains 匹配”，还不是一个语义明确、可预期的检索入口。

当前行为可以概括为：

- 输入内容写入路由 query `cfq`
- 前端以 280ms debounce 同步到 URL
- 列表刷新时把 `q` 传给 `GET /api/curated-frames`
- Web API 模式下由后端做 `LIKE '%q%'`
- Mock 模式下由前端本地 haystack 做 `includes`

当前可命中的字段包括：

- `title`
- `code`
- `movieId`
- `actors`
- `tags`
- `capturedAt`
- `positionSec`
- 以及格式化后的时间串（如 `01:23`）

这说明它的优点是“成本低、覆盖广”，但也带来几个明显问题：

- 用户不知道自己到底在搜“影片编号 / 演员 / 标签 / 时间点”中的哪一种
- 同一个搜索框同时承担全文检索、编号直达、时间定位、标签查找，语义过宽
- 标签 facet 不跟随搜索词联动，用户输入后顶部标签仍像“全库标签”，上下文感不够强
- 后端当前是拼接长字符串后 `LIKE`，召回宽，但排序与可解释性弱
- Mock 与 Web API 虽然大体一致，但细节仍可能出现边缘不一致

### 2. 我认为最值得优先优化的，不是“更模糊”，而是“更明确”

如果只能优先做一批，我建议先把搜索框从“一个模糊的万能框”升级成“一个有明确检索提示、轻度结构化、结果更稳定的入口”。

优先级建议如下：

1. 先优化搜索语义提示与命中反馈
2. 再优化搜索结果排序
3. 最后再考虑高级语法或多字段结构化检索

原因很简单：

- 现在的最大问题不是搜不到，而是用户不确定“为什么搜到这些”
- 只增加模糊匹配或分词，往往会让结果更散
- 先把搜索的意图和结果解释清楚，收益比直接上复杂语法更高

### 3. 第一阶段最划算的优化

#### 方案 A：补强搜索框提示文案

当前 placeholder 如果只是“搜索萃取帧”，信息密度偏低。

建议改成更明确的提示，例如：

- `搜索编号、标题、演员、标签、时间`
- 或 `支持编号 / 演员 / 标签 / 时间点搜索`

这类改动成本很低，但能立刻降低误用率。

#### 方案 B：把当前搜索上下文显示出来

当用户输入后，结果区顶部建议补一句上下文提示，例如：

- `搜索：三上`
- `命中 48 帧 · 12 片`
- `当前标签：构图；当前搜索：制服`

这样用户滚动很久后，仍然知道结果是按什么上下文筛出来的。

#### 方案 C：给结果排序加入“强命中优先”

当前后端主要是 `captured_at DESC`，这对“最新浏览”友好，但对搜索不够友好。

建议在有 `q` 时切换成轻量排序优先级，而不是永远只按时间倒序：

1. `code` 精确命中优先
2. `title` / `actor` / `tag` 前缀或完整词命中优先
3. 其余全文 contains 命中
4. 同层级内再按 `capturedAt DESC`

这样可以明显改善“搜番号时最相关结果不在前面”的问题。

### 4. 第二阶段建议：把搜索框做成“轻结构化单框”

我不建议立刻拆成多个输入框，但很建议支持少量轻语法，让高级用户更稳定地收窄范围。

推荐支持这些最小语法：

- `tag:构图`
- `actor:三上`
- `code:ABP-123`
- `time:01:23`

同时保留裸词搜索：

- 裸词仍走全文检索
- 带前缀的词走定向字段检索

这样有几个好处：

- 保留一个输入框，不破坏页面简洁度
- 高级用户可以稳定表达意图
- 后端查询可以逐步从“大字符串 LIKE”演进到“字段级过滤”

这一步比直接做“多输入框筛选条”更适合当前页面阶段。

### 5. 第三阶段建议：让 facet 与搜索联动

这是我认为体验提升很大的点。

当前标签区更像全库 facet；用户输入搜索词后，标签统计最好切换成“当前搜索结果上下文下的 facet”。

例如用户搜 `三上` 后，顶部标签不要继续显示全库的：

- 制服 120
- 构图 98

而应该显示“在三上的结果集合里”的：

- 制服 17
- 构图 9
- 近景 8

这样搜索框和标签区才是协同关系，而不是两个彼此独立的过滤器。

如果后端后续要演进，建议让：

- `GET /api/curated-frames/tags`

支持可选上下文参数：

- `q`
- `actor`
- `movieId`

这样 facet 才能真正成为“上下文感知”的 facet。

### 6. 不建议优先做的方向

有几类优化看起来高级，但我不建议优先上：

- 不建议先上复杂分词或搜索引擎式模糊匹配
- 不建议先上多标签 AND / OR 查询语法
- 不建议先把搜索框拆成多个并列字段输入框
- 不建议先做横向滚动的高级筛选工具条

原因是当前页面的核心问题仍然是：

- 搜索意图不够明确
- 结果相关性不够稳定
- 搜索与标签上下文没有联动

这些基础问题没处理好，继续堆功能只会增加复杂度。

### 7. 我的最终建议排序

如果你问我“当前这个搜索框最值得怎么优化”，我的排序是：

1. 先补清楚 placeholder 和结果区上下文反馈
2. 再做搜索结果的相关性排序，而不是一律按 `capturedAt DESC`
3. 再支持少量前缀语法：`tag:` / `actor:` / `code:` / `time:`
4. 最后让标签 facet 跟随当前搜索词联动

如果只做一个最小但值回票价的改动，我会选：

- `搜索结果排序优化 + 当前搜索上下文显示`

因为这会直接提升“搜出来的结果像不像我想找的东西”，比单纯把搜索框做得更花更复杂更有价值。

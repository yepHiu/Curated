# 首页 / 详情 / 侧栏统一优化方案草案

## 背景

用户希望把以下 3 个体验问题一起优化掉：

1. 首页在进入详情页或其他页面后，返回时恢复上下滚动位置
2. 影片详情页简介过长时，默认折叠，支持展开 / 收起
3. 当影片数量变多后，桌面侧栏折叠 / 展开出现卡顿，需要给出性能优化方案

本方案只覆盖这 3 个点，不顺带扩散到全局滚动恢复、整页虚拟化改造或导航体系重构。

---

## 当前实现观察

### 1. 首页滚动恢复

- `HomeView.vue` 只是把数据传给 `HomepagePortal.vue`
- `HomepagePortal.vue` 自己持有滚动根：`data-home-scroll-region` + `overflow-y-auto`
- 当前首页没有独立的滚动恢复 composable
- 仓库中已有两种相关先例：
  - `useLibraryScrollPreserve.ts`：会主动保存 / 恢复滚动位置
  - `use-settings-scroll-preserve.ts`：此前因为实现过于激进，已被降级为 no-op

结论：

- 首页可以做“局部滚动恢复”
- 但不能复用一个过于通用、侵入式、带多次补偿写回的全局方案

### 2. 详情页简介

- `DetailPanel.vue` 中 `summaryDisplay` 已经被格式化为纯文本展示
- 当前直接整段渲染：
  - `v-if="summaryDisplay"`
  - `class="text-pretty text-sm leading-6 text-muted-foreground"`
- 没有“展开 / 收起”状态，也没有行数裁剪

结论：

- 这个点是纯前端展示层问题
- 不需要改 DTO / API / 数据模型

### 3. 侧栏折叠卡顿

- `AppShell.vue` 通过切换 `grid-template-columns` 做桌面侧栏宽度过渡
- `AppSidebar.vue` 在 `compact` 和非 `compact` 间切换时，会切换两套明显不同的 DOM 分支：
  - 展开态：完整标题、Browse / Yours 分组、`ScrollArea`、计数 badge、后端状态卡片、Settings 文本按钮
  - 折叠态：仅图标导航、分隔线、状态点、Settings 图标按钮
- 当前折叠不是“同一棵树轻量变体”，而是“整块内容重建 + 外层列宽动画”并发发生
- 当影片数量变多时，`sidebarNavGroups` 的派生统计也会在切换时一起参与重算

结论：

- 卡顿的根因更像“布局动画 + 分支卸载/重建 + 统计派生耦合”
- 不是单纯 CSS duration 太长

## 可选方案

### 方案 A：最小改法，逐点补丁

做法：

- 首页单独加一个轻量滚动快照
- 详情页简介直接加 `line-clamp` + 展开按钮
- 侧栏仅缩短动画时长、减少阴影和过渡属性

优点：

- 改动小
- 风险最低
- 交付最快

缺点：

- 侧栏卡顿只能缓解，不能真正解决 DOM 切换成本
- 后续仍可能在大库数据下复发

### 方案 B：推荐，做“局部状态持久化 + 轻量展示折叠 + 侧栏结构降切换成本”

做法：

- 首页只为 `data-home-scroll-region` 增加专用滚动恢复，不做全局推广
- 详情简介做可测的折叠组件 / 逻辑，只有超过阈值时才显示“展开”
- 侧栏保留现有导航信息与视觉结构，但把“展开态 / 折叠态”改成更接近同构的两层：
  - 导航数据与计数派生稳定
  - DOM 尽量保留
  - 折叠时优先隐藏文本和 badge，而不是整段拆两套

优点：

- 体验提升更稳定
- 风险仍可控
- 不需要重做 AppShell 结构

缺点：

- 改动比方案 A 大
- 需要补测试覆盖 3 条链路

### 方案 C：偏重性能重构

做法：

- 统一引入页面滚动缓存层
- 详情页拆出专用 `ExpandableText`
- 侧栏彻底改造成单树渲染 + CSS-only collapse + 统计预计算缓存

优点：

- 长期最干净
- 侧栏性能上限最高

缺点：

- 范围偏大
- 容易超出这次需求边界
- 验证成本明显上升

---

## 推荐方案

推荐 **方案 B**。

原因：

- 它能把 3 个点一起解决，但仍然把范围控制在当前页面和壳层内
- 首页滚动恢复保持“局部、克制、可回退”
- 详情简介属于标准展示增强，收益高、风险低
- 侧栏卡顿如果只调动画参数，收益不够；但如果直接做全重构，范围又过大

## 设计方案

## 一、首页滚动恢复

### 目标

- 从首页进入详情、播放页、资料库筛选页后，再回到首页时，尽量恢复用户离开前的纵向滚动位置
- 只影响首页滚动容器，不影响 AppShell 其他页面

### 方案

- 新增一个首页专用 composable，例如：
  - `use-home-scroll-preserve.ts`
- 只绑定 `HomepagePortal.vue` 的 `data-home-scroll-region`
- 使用一个稳定 key 保存首页滚动位置，例如固定 key：
  - `home:portal`
- 保存策略：
  - 在滚动事件中节流保存 `scrollTop`
  - 在组件卸载前再保存一次
- 恢复策略：
  - 仅在首页组件挂载完成后恢复一次
  - 使用克制的恢复序列：
    - `nextTick`
    - 一次 `requestAnimationFrame`
    - 最多再补一次短延时回写
  - 不采用 `useLibraryScrollPreserve` 当前那种多段强补偿链

### 边界

- 首页内容结构变化很大时，允许恢复位置不完全精确
- 不把这套逻辑推广到 settings、detail、history 等其他页
- 若快照不存在，则保持当前默认从顶部开始

### 验证重点

- 首页滚动到中段后进入详情，再返回时恢复到接近原位置
- 首次进入首页不应被错误滚动
- Daily recommendations 刷新 / portal 数据重算时不应频繁抢滚动

---

## 二、详情页长简介折叠

### 目标

- 简介短时保持当前直接展示
- 简介长时默认折叠，用户可展开查看全文，再可收起

### 方案

- 在 `DetailPanel.vue` 内引入一个轻量折叠展示层，推荐抽成小组件：
  - `ExpandableText.vue`
  - 或先在 `DetailPanel.vue` 内实现同等逻辑
- 基础行为：
  - 默认最多展示 4 到 6 行，推荐 **5 行**
  - 超过阈值时显示：
    - “展开简介”
    - 展开后显示“收起简介”
- 判定方式：
  - 先用文本长度阈值做快速判定，避免首屏直接依赖 DOM 测量
  - 可在挂载后补一次真实行高/溢出判定，修正边界案例

### 视觉建议

- 折叠态保留当前正文样式
- 底部增加轻量渐隐，不做重卡片包裹
- 按钮用 `ghost/link-like` 风格，避免喧宾夺主

### 边界

- 只有 `summaryDisplay` 非空时才渲染
- 很短的简介不显示展开按钮
- 展开状态仅在当前详情页会话内保留，不跨影片复用

### 验证重点

- 短简介不出现多余按钮
- 长简介默认折叠，点击后能完整展开
- 切换到另一部影片时折叠状态重置

---

## 三、侧栏折叠卡顿优化

### 目标

- 改善桌面侧栏从展开到折叠、再从折叠到展开时的卡顿
- 保持现有导航结构、分组、计数、后端状态和设置入口

### 问题拆解

当前卡顿大概率来自三件事叠加：

1. `AppShell` 的 `grid-template-columns` 动画会触发布局重算
2. `AppSidebar` 在 `compact` 切换时会卸载 / 重建大块 DOM
3. 侧栏统计信息在切换期间也会参与派生计算

### 方案

#### 3.1 保持导航数据稳定，避免折叠时重新组织大块结构

- `sidebarNavGroups` 继续保留，但把统计派生和导航项映射尽量放到稳定 computed 中
- 折叠切换时不改变数据结构，只改变展示方式

#### 3.2 从“双分支切换”调整为“单树优先，文本隐藏为主”

- 展开态和折叠态尽量复用同一套导航 item DOM
- 图标节点稳定保留
- 标签文字、badge、section title、后端 meta 文本在折叠时使用：
  - `opacity`
  - `max-width`
  - `overflow-hidden`
  - `pointer-events-none`
  - 条件 `aria-hidden`
- 尽量避免 `v-if` 把整段 section / ScrollArea 直接拆成另一棵树

#### 3.3 缩窄动画范围

- `AppShell` 仍可保留列宽过渡，但时长可从 `300ms` 收紧到更利落的区间，例如 `200ms~240ms`
- `AppSidebar` 避免对 `padding`、大量颜色、阴影同时做过渡
- 过渡重点放在：
  - 外层宽度
  - 文本透明度
  - 少量 transform

#### 3.4 把高频无关区块从折叠主动画里解耦

- 后端状态区域、badge 文本等非导航核心内容可在折叠时做“延后淡出 / 淡入”
- 避免和主导航宽度动画完全同步，减少同帧压力

### 不建议这次做的事

- 不重做为完全虚拟化侧栏
- 不把整个侧栏改成复杂的 FLIP 动画系统
- 不修改导航路由结构

### 验证重点

- 大库数据下连续点击折叠 / 展开，肉眼卡顿明显下降
- 折叠时焦点、hover、active 路由状态不丢
- 桌面与移动抽屉逻辑不互相影响

---

## 测试建议

### 首页

- 为 `HomeView` / `HomepagePortal` 增加滚动恢复测试
- 至少覆盖：
  - 记录滚动位置
  - 重新挂载后恢复
  - 无快照时不误恢复

### 详情页

- 为简介折叠逻辑增加组件测试
- 至少覆盖：
  - 短简介不显示折叠按钮
  - 长简介默认折叠
  - 点击展开 / 收起切换正常

### 侧栏

- 为 `AppSidebar` / `AppShell` 增加结构测试
- 至少覆盖：
  - 折叠态不改变核心导航项数量
  - 折叠按钮切换后关键节点仍存在
  - 不再依赖两套完全分离的主导航 DOM

---

## 建议实施顺序

1. 首页滚动恢复
2. 详情简介折叠
3. 侧栏折叠性能优化

原因：

- 前两项更独立，能先稳定交付
- 侧栏性能优化依赖先确认当前壳层行为和测试基线

---

## 成功标准

- 首页返回时滚动位置基本恢复，不再每次从顶部开始
- 长简介默认不“顶满屏”，展开 / 收起明确且自然
- 大量影片数据下侧栏折叠 / 展开肉眼卡顿明显降低
- 不引入新的整页白屏、滚动乱跳或导航状态异常

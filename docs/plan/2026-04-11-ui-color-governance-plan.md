# 2026-04-11 Curated UI 颜色治理方案草案

## 状态

- 状态：In Progress
- 主题：收敛 Curated 前端颜色使用边界，降低页面间状态色和表面色不一致问题

## 背景判断

当前项目并不是“没有颜色系统”，而是“有 token，但消费边界不严格”。

已存在的基础：

- `src/style.css` 已定义全局语义色：`primary`、`success`、`warning`、`danger`、`info`、`background`、`card`、`muted`、`border` 等
- `docs/reference/2026-03-24-frontend-ui-spec.md` 已要求业务组件优先使用语义化 token

当前问题：

- 多个业务页面直接使用 Tailwind 原生色阶，如 `amber-*`、`emerald-*`、`red-*`、`slate-*`
- 同一语义在不同页面的表现不一致，例如“警告”有时偏浅黄提示块，有时是深黄文字，有时是橙黄 badge
- 少数区域还存在直接硬编码颜色或专属色板，导致 UI 呈现出“同项目、不同页面像不同作者写的”

## 目标

- 明确哪些颜色必须通过语义 token 使用
- 明确哪些区域允许例外
- 减少业务组件直接写 Tailwind 色阶
- 把“成功 / 警告 / 危险 / 信息”统一成稳定的视觉语义
- 优先提升一致性，不为了治理而重做整站视觉

## 不做的事

- 不在第一轮重做品牌主色
- 不在第一轮重刷所有页面和所有组件
- 不强行把播放器沉浸层、开发环境标记层完全改造成普通业务页风格
- 不为了颜色治理同时引入大规模布局或排版改造

## 方案选项

### 方案 A：只补文档，不改代码

做法：

- 补一份颜色使用规范
- 后续开发时人工遵守

优点：

- 成本最低
- 不影响现有页面

缺点：

- 现有不一致会继续存在
- 团队很难靠记忆长期执行

### 方案 B：轻量治理，只统一状态语义和高频 raw color

做法：

- 保留现有主题 token
- 定义颜色使用边界
- 优先治理成功 / 警告 / 危险 / 信息四类状态
- 优先处理设置页、萃取帧页、开发条等高曝光差异点

优点：

- 风险低
- 收益直接
- 能快速形成统一感

缺点：

- 仍会保留一部分历史页面差异
- 需要第二轮继续收尾

### 方案 C：完整规范化，组件和页面全部改为 token-only

做法：

- 统一替换业务组件中的直接色阶
- 为 Badge / Alert / StatusDot / StatusPanel 增加系统化 variant
- 页面层只允许使用语义色

优点：

- 最干净
- 后续维护成本最低

缺点：

- 改动面大
- 容易误伤已有视觉语义
- 当前阶段投入产出比偏低

## 推荐方向

推荐采用 **方案 B，带少量方案 C 的基础设施准备**。

也就是：

- 第一轮不追求把所有页面改干净
- 先把“颜色语义不一致”最明显的部分收敛
- 同时补足可复用的状态视觉承载方式，为后续完整规范化打基础

这是当前最务实的路径。

## 颜色治理规则

### 1. 基础表面色规则

普通业务页默认只允许使用以下语义色：

- `background`
- `foreground`
- `card`
- `surface`
- `muted`
- `accent`
- `border`
- `primary`

这些用于：

- 页面背景
- 卡片表面
- 默认文字
- 边框
- 主按钮和主强调

业务页不应直接写 `slate-*`、`zinc-*`、`gray-*` 作为主要表面或文本颜色。

### 2. 状态色规则

状态视觉统一只使用四类语义：

- 成功：`success`
- 警告：`warning`
- 危险：`danger` / `destructive`
- 信息：`info`

业务组件不再直接写：

- `emerald-*`
- `amber-*`
- `red-*`
- `blue-*`

除非它属于被明确标记的例外区域。

### 3. 例外区域规则

以下区域允许保留局部专属颜色：

- 播放器沉浸层 HUD
- 开发环境角标 / 性能监视
- 媒体内容覆盖层

这些区域可以使用专属深色、白色透明度和局部警示色，但要满足两个约束：

- 例外只存在于该局部，不向普通业务页扩散
- 同一例外区域内部仍应自洽，不出现多套状态色

## 实现落点建议

### 第一阶段：规则补齐

补一份明确的颜色治理文档，并回写到现有 UI 规范文档中，至少明确：

- 允许使用的基础语义色
- 状态色语义映射
- 允许例外的区域
- 禁止在业务组件中直接写原生 Tailwind 状态色阶

### 第二阶段：组件承载补齐

补足或扩展以下承载方式：

- `Badge` 的 `success / warning / info` 变体
- 状态提示块的统一 class 组合，或抽出 `StatusPanel`
- 状态圆点 / 状态文本的统一映射函数

目标不是抽很多组件，而是让高频场景不再每页手写一套颜色。

### 第三阶段：优先治理高差异页面

优先处理以下位置：

- `SettingsPage.vue`
- `CuratedFramesLibrary.vue`
- `DevPerformanceBar.vue`
- `DevEnvironmentBadge.vue`
- `AppSidebar.vue`

原因：

- 它们直接暴露了最明显的颜色分裂
- 覆盖了普通业务页、状态页和开发辅助层三类典型场景

### 第四阶段：保留播放器例外，但收敛边界

播放器页可继续保留沉浸式深色浮层，但要把：

- 警告文本
- 错误提示
- 面板边框和文字层级

统一到一套播放器内部规则，而不是继续零散拼色。

## 我建议的执行顺序

1. 先定规则，不先大面积换色
2. 先补状态色承载，不先改所有组件
3. 先治理高曝光页面，不先全项目扫描替换
4. 最后再决定是否做完整 token-only 收敛

## 当前结论

Curated 现在最缺的不是“更多颜色”，而是“颜色使用权限”。

也就是：

- 哪些颜色是全局语义
- 哪些颜色只能做状态
- 哪些区域允许例外
- 哪些业务组件不能再随手写原生色阶

只要先把这层边界收紧，整体 UI 一致性会明显提升，而且不会把当前产品的可用性和节奏打乱。

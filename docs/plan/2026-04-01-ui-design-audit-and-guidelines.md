# Curated UI 设计审计与规范草案

## 1. 文档目的

本文档用于沉淀当前 Curated 前端的：

- 当前设计语言判断
- 目标设计方向
- 设计要求
- 设计原语
- 页面级规范
- 统一化改造建议

## 2. 当前 UI 的设计语言判断

当前 Curated 的 UI 不是“缺少审美”，而是处于一个典型的中期阶段：

- 壳层开始形成统一语言
- 设置页已经有较强规范化倾向
- 资料库页承担核心体验，但结构层级仍偏重
- 演员页、历史页、精选帧页和资料库页之间还没有共享同一套页面语法

可以把当前风格概括为：

- 基础方向：`content-first media workspace`
- 结构气质：`桌面应用式壳层 + shadcn 风格表面`
- 当前问题：`页面各自成立，但不像同一设计系统的产物`

更具体地说，当前界面混合了四种不同的页面语言：

- 壳层语言：侧边栏 + 顶部工具条 + 主工作区
- 浏览页语言：筛选条 + tab + 海报网格
- 设置页语言：section nav + form cards + action rows
- 列表页语言：窄内容流或纯网格

这些语言单独看都不是错，但并行存在时会削弱“统一化”和“规范化”的感知。

## 3. 参考目标的设计方向

参考图来自 shadcn-vue blocks 的 dashboard shell。我们这次不借用它的颜色、品牌和具体业务模块，只吸收它的空间组织思想。

目标方向应定义为：

- `Calm Workspace`
- `Content-first`
- `Low chrome`
- `Dense but readable`
- `Structural consistency over decoration`

翻译成产品界面要求，就是：

- 壳层明确，页面像“同一个应用里的不同工作区”
- 每页只有一个主任务区
- 减少同等级的大块卡片互相竞争
- 让筛选、导航、状态、内容之间的主次关系稳定
- 统一圆角、边框、宽度和标题体系

## 4. 设计要求

### 4.1 全局要求

- 全站应围绕 `AppShell` 建立统一工作区感。
- 所有页面必须共享同一套外层节奏：页边距、内容宽度、标题层级、容器间距。
- 禁止每个页面独立发明一套“容器风格”。
- 优先靠布局、留白、尺寸和层级建立高级感，而不是堆阴影、堆边框、堆颜色变化。

### 4.2 页面组织要求

- 每个页面只能有一个主任务区。
- 页面顶部只能有一个核心控制带。
- 次级信息必须从属于主内容，而不是和主内容争夺视觉中心。
- 页面内部的“筛选提示”“当前状态”“空态”“错误态”需要共享统一组件语法。

### 4.3 shadcn 约束

- 优先复用现有 `Button`、`Card`、`Tabs`、`Badge`、`Separator`、`ScrollArea` 等原语。
- 允许通过 class 调整布局，不应到处发明新的视觉组件类型。
- 同类功能应复用同一种 shadcn 组合方式。

## 5. 当前最核心的问题

### 5.1 页面容器语法不统一

不同页面分别使用：

- 大卡片包裹
- 窄列阅读流
- 纯网格流
- 卡片套卡片的管理台

结果是：

- 用户能看出这是一个项目
- 但看不出这是一套完整设计系统

### 5.2 主次关系不稳定

资料库页在不同模式下容易出现多个“看起来都很重要”的区域：

- 筛选提示
- 标签浏览卡片
- tabs
- 批量操作
- 海报工作区

这会让页面显得“内容很多”，但不显得“结构清楚”。

### 5.3 设置页过于成熟，其它页跟不上

设置页的规范完成度明显高于演员页、历史页、部分浏览页：

- 有明确 section nav
- 有统一卡片头
- 有一致的操作行模式

这说明项目已经自然长出了设计规范雏形，但还没有抽象成全站规则。

### 5.4 宽度策略不统一

当前不同页面的内容宽度策略差异较大：

- 有的完全铺满工作区
- 有的限制到 `max-w-4xl`
- 有的又是偏宽的 gallery 容器

这不是不能有差异，而是缺乏命名和规则。

## 6. 建议沉淀的设计原语

这部分是最值得优先落地的内容。

### 6.1 页面原语

建议统一出以下页面级原语：

#### `PageHeader`

职责：

- 页面身份
- 标题
- 副标题
- 可选右侧操作

规范：

- 页面标题字号固定
- 副标题固定为弱说明文案
- 操作区最多 1 组，不与主内容竞争

适用页面：

- 历史页
- 演员页
- 设置页分段顶部
- 资料库特殊模式页

#### `PageToolbar`

职责：

- tabs
- 视图切换
- 批量管理
- 页面级筛选动作

规范：

- 一页只允许一个主要 toolbar
- toolbar 在视觉上必须低于页面标题、高于内容区
- 不能同时出现多个类似 toolbar 的条带

#### `PageSection`

职责：

- 标准内容容器

规范：

- 用于承载正文内容区、统计区、筛选区、表单区
- 统一圆角、边框、内边距
- 不在不同页面里随意改成完全不同的表面语言

#### `InlineFilterBar`

职责：

- 展示“当前筛选条件”
- 提供清除入口

规范：

- 全站统一样式
- 不因页面不同变成不同高度、不同圆角、不同信息结构

#### `EmptyStateBlock`

职责：

- 空数据、空历史、空筛选结果

规范：

- 统一标题、说明、主动作布局
- 统一使用中性表面，不临时造新视觉样式

### 6.2 表单原语

建议将设置页里的行结构沉淀为：

- `SettingCard`
- `SettingRow`
- `SettingField`
- `SettingStatusHint`

设置页已经接近这个结构，只是目前还写在大模板中，未提炼为页面规则。

## 7. 建议统一的设计 Token 规则

### 7.1 圆角

建议只保留四档：

- `shell`: 页面壳层
- `section`: 标准内容容器
- `control`: 表单与按钮
- `chip`: badge、tag、筛选粒子

不要继续出现大量“这里 1.5rem、那里 2rem、那里 3xl”的自由增长。

### 7.2 宽度

建议统一三套页面宽度模式：

- `reading`
  - 历史、关于、说明类
- `workspace`
  - 资料库、演员、精选帧
- `form`
  - 设置

页面只选择宽度模式，不自行发明 `max-w-*`。

### 7.3 标题层级

建议统一为：

- `Page Title`
- `Section Title`
- `Body`
- `Meta`

避免不同页面标题看起来像来自不同产品。

### 7.4 边框和阴影

建议只保留：

- 结构边框
- 弱分隔边框
- 浮层阴影
- 强调阴影

常规页面不要依赖阴影制造层级，优先使用留白和结构边界。

## 8. 页面级规范建议

### 8.1 AppShell

应继续维持：

- 左侧固定导航
- 中央顶部工具条
- 下方主工作区

接下来还可继续优化：

- 右上角全局动作区从“只有主题切换”升级为统一 action slot
- 左侧返回区和当前页面身份区的规则进一步稳定

### 8.2 资料库页

定位：

- 核心主场景
- 一切设计语言优先围绕它来建立

建议：

- 普通浏览模式只保留一个主 toolbar
- tags 模式不要让标签卡片压过海报内容区
- 过滤状态条、tabs、批量操作统一为同一条工作带中的不同槽位
- 海报区继续作为唯一主舞台

### 8.3 演员页

当前问题：

- 页面骨架偏轻
- 更像一个“有标题的网格页”

建议：

- 接入和资料库页一致的 `PageHeader + PageSection`
- 网格区包进统一 `PageSection`
- 筛选条与空态改用统一 `InlineFilterBar / EmptyStateBlock`

### 8.4 历史页

当前问题：

- 阅读流感很强，但与主产品“工作区”语言脱节

建议：

- 保留窄列优势
- 但仍应接入统一的 `PageHeader`
- day bucket 区域建议拥有更稳定的 section 分段规则
- 空态采用与演员页、资料库一致的空态语法

### 8.5 设置页

当前评价：

- 是项目里最接近“规范模板”的页面

建议：

- 把设置页当作规范来源，而不是特殊页面
- 将卡片头、行结构、section nav 的规则反向提炼给其它页面复用

## 9. 优化建议优先级

### Phase 1：先做系统统一

- 抽出 `PageHeader`
- 抽出 `PageSection`
- 抽出 `InlineFilterBar`
- 定义 3 套页面宽度模式
- 定义圆角和标题层级 token

这是收益最大的一步。

### Phase 2：统一浏览页家族

- 重构资料库页上半区层级
- 让演员页接入统一页面骨架
- 让历史页接入统一标题与 section 规则
- 让精选帧页与资料库页拥有更接近的工作区语法

### Phase 3：统一设置页与全局动作区

- 将设置页已有模式抽象成全站规范
- 为 AppShell 顶栏建立右侧 action slot
- 统一页面内状态提示、成功提示、错误提示的写法

## 11. 本轮结论

Curated 当前最需要的不是“更多设计”，而是“更少但更一致的设计”。

接下来的优化重点不应是：

- 单个页面再美化一点
- 多加一些卡片
- 多加一点视觉强调

而应是：

- 抽象统一原语
- 让不同页面说同一种界面语言
- 让资料库页成为主场景标准
- 让设置页成为规范来源

只要完成这一步，整体专业感会明显提升，而且后续再加新页面也不会继续发散。

# Curated 页面语言重构方案

## 1. 目标

本方案用于将 Curated 当前“页面各自成立但缺少统一页面语言”的状态，重构为一套可复用、可扩展、可持续维护的页面设计系统。

本次重构不追求一次性重做所有视觉，而是优先建立统一页面语法，让后续任何页面优化都建立在同一套语言上。

核心目标：

- 统一页面骨架
- 统一主次层级
- 统一容器语法
- 统一宽度与间距规则
- 统一状态块与筛选块的表达方式
- 让“资料库页”成为主场景标准，让“设置页”成为规范来源

## 2. 当前问题总结

当前问题不是单个页面“设计差”，而是整套 UI 同时存在多种页面语言：

- 壳层页语言：侧边栏 + 顶栏 + 工作区
- 浏览页语言：筛选条 + tab + 海报网格
- 设置页语言：section nav + setting card + action row
- 列表页语言：窄内容流 / 单列列表 / 纯网格

这些页面语言都可以成立，但缺少统一抽象，导致：

- 页面之间切换时像进入不同子产品
- 同类功能使用不同容器表达
- 主任务区不够稳定
- 设计规范无法沉淀，新增页面会继续发散

## 3. 重构原则

### 3.1 先统一语法，再优化视觉

先解决：

- 页面结构是否一致
- 内容分区是否有规则
- 主次关系是否稳定

再解决：

- 某个页面还要不要更精致
- 哪个边框更细
- 哪种卡片更好看

### 3.2 页面只允许有限的布局模式

不要继续让每个页面自由定义自己的宽度、间距、容器和标题层级。

未来页面只能从预定义模式中选择，不再“临时拼一页”。

### 3.3 页面原语优先于业务块

优先抽象：

- 页面头部
- 标准内容容器
- 筛选状态条
- 空态

而不是先抽象业务块。

原因是：

- 页面原语能统一所有页面
- 业务块只能服务局部

### 3.4 主场景优先

Curated 的主场景是浏览、筛选、查看、播放。

因此所有页面语言都应优先服务：

- 内容检索
- 内容浏览
- 状态切换
- 工作流连续性

而不是把产品做成“到处都是管理卡片”的后台。

## 4. 目标页面语言

建议将 Curated 的页面语言定义为：

- `Desktop-first media workspace`
- `Low-chrome structured application`
- `Content-first with calm surfaces`

对应到体验层面：

- 壳层稳定
- 内容区是主舞台
- 页面头部统一
- 次要信息服从主工作区
- 辅助块尽量轻，不抢主内容

## 5. 需要建立的页面原语

本次重构最核心的输出不是单页改动，而是以下页面原语。

### 5.1 `PageLayout`

职责：

- 承载页面内部整体节奏
- 统一页面顶部、主内容、底部保留空间关系

规范：

- `flex flex-col`
- 统一纵向 gap
- 统一最外层宽度约束接入方式
- 页面内部不随意再包“无意义大卡片”

适用：

- 所有 `RouterView` 页面

### 5.2 `PageHeader`

职责：

- 页面身份
- 标题
- 副标题
- 右侧操作

规范：

- 标题永远是页面最强文本锚点
- 副标题永远是弱说明
- 右侧操作数量受控
- 不与 toolbar 混写

适用：

- Actors
- History
- Curated Frames
- Settings 分节页面
- 特殊模式的 Library

### 5.3 `PageToolbar`

职责：

- tabs
- 批量管理
- 视图控制
- 页面级筛选动作

规范：

- 一页最多一个主要 toolbar
- toolbar 在页面标题之后、内容之前
- toolbar 与主内容有稳定间距
- 同级按钮密度一致

适用：

- Library
- Curated Frames
- 未来的任何浏览页

### 5.4 `PageSection`

职责：

- 标准内容容器

规范：

- 统一圆角
- 统一边框强度
- 统一内边距
- 统一标题和内容间距

用途：

- 标签区
- 网格区
- 统计区
- 表单分区
- 列表区

### 5.5 `InlineFilterBar`

职责：

- 当前筛选说明
- 清除筛选动作

规范：

- 统一高度节奏
- 统一文案结构
- 统一清除按钮尺寸和位置

用途：

- Library actor/tag/studio 过滤提示
- Actors tag 过滤提示
- 未来任意页面的 active filter 状态

### 5.6 `EmptyStateBlock`

职责：

- 空数据
- 空筛选结果
- 空历史

规范：

- 标题、说明、行动按钮布局统一
- 视觉语气中性
- 不为单个页面发明一套新空态

### 5.7 `SettingSectionCard`

职责：

- 设置页标准分组容器

规范：

- 继承 `PageSection`
- 补充图标头、描述区、内容行规范

### 5.8 `SettingRow`

职责：

- 单个设置项的三段式表达

规范：

- 左：标题与说明
- 右：控件
- 变体：开关行、输入行、双列行、危险操作行

## 6. 页面宽度与布局模式

未来所有页面只允许从以下三种模式中选，不再手写新的宽度策略。

### 6.1 `workspace`

适用：

- Library
- Actors
- Curated Frames

特征：

- 主内容铺满工作区
- 更适合网格、面板、批量操作
- 顶部控制带更明显

### 6.2 `reading`

适用：

- History
- About
- 说明型页面

特征：

- 内容宽度收敛
- 强调阅读与纵向扫描
- 页面头和内容节奏更克制

### 6.3 `form`

适用：

- Settings

特征：

- 允许分栏导航
- 内容区偏中等宽度
- 强调连续设置操作与字段阅读

## 7. 视觉 Token 重构建议

本次不一定全部立刻改，但要先定义规则。

### 7.1 圆角分层

建议只保留四档：

- `radius-shell`
- `radius-section`
- `radius-control`
- `radius-chip`

### 7.2 间距分层

建议只保留稳定的页面间距梯度：

- 页面级间距
- section 间距
- 控件组间距
- 微间距

禁止每个页面再独立增长自己的 spacing 语言。

### 7.3 标题层级

建议固定：

- `page-title`
- `section-title`
- `body`
- `meta`

### 7.4 表面层级

建议固定：

- 壳层表面
- 标准 section 表面
- 内嵌弱表面
- 浮层表面

避免页面内部出现多层“看起来都像主卡片”的表面竞争。

## 8. 按页面的重构策略

### 8.1 AppShell

目标：

- 成为稳定全局框架，不再继续承担单页视觉补丁角色

改造点：

- 保持左右壳层结构
- 顶栏抽象出稳定的三段布局
- 给右上角预留统一 action slot
- 保证所有页面接入统一内容画布

### 8.2 Library

目标：

- 成为主场景标准页

改造点：

- 普通浏览模式：`PageToolbar + PageSection(content)`
- `tags` 模式：弱化标签大卡片，避免压制内容区
- 所有 filter 状态统一接入 `InlineFilterBar`
- 网格区维持唯一主舞台地位

### 8.3 Actors

目标：

- 从“标题 + 网格页”升级为完整 workspace 页

改造点：

- 接入 `PageHeader`
- 网格区接入 `PageSection`
- 筛选条与空态统一原语
- 与 Library 共享浏览页语法

### 8.4 History

目标：

- 保留阅读感，但纳入统一页面系统

改造点：

- 接入 `PageHeader`
- day bucket 有统一 section 规则
- 空态与其它页面一致
- 控制阅读宽度，但不脱离工作区语言

### 8.5 Curated Frames

目标：

- 与 Library 共享同一类工作区模型

改造点：

- 对齐 `PageToolbar`
- 对齐 section 容器
- 对齐批量操作表达

### 8.6 Settings

目标：

- 从“成熟独立页”转为“系统规范样板页”

改造点：

- 将现有成熟结构拆出通用原语
- 收敛重复 card 样式
- 将 setting row 抽象为固定布局模式

## 9. 组件落地建议

建议新增目录：

- `src/components/jav-library/page/`

- `PageLayout.vue`
- `PageHeader.vue`
- `PageToolbar.vue`
- `PageSection.vue`
- `InlineFilterBar.vue`
- `EmptyStateBlock.vue`
- `SettingSectionCard.vue`
- `SettingRow.vue`

## 10. 实施阶段

### Phase 1：建立原语，不动业务逻辑

目标：

- 只新增页面原语与 token 约束
- 不改变服务层和数据流

输出：

- 页面原语组件
- 文档更新

### Phase 2：迁移三类页面

优先顺序：

1. Library
2. Actors
3. History

原因：

- 这三页最能提升整体统一感

### Phase 3：整理 Settings 规范来源

目标：

- 把 Settings 的成熟模式抽象为公共规则

### Phase 4：全局微调

目标：

- 收敛圆角
- 收敛标题层级
- 收敛 section 间距
- 收敛边框强度

## 11. 验收标准

页面语言重构完成后，应满足：

- 任意页面都能看出属于同一产品
- 页面头部结构一致
- 主工作区位置稳定
- 状态条、空态、section 容器表达一致
- 页面宽度选择有规则
- 新页面不需要重新发明布局语法

## 12. 建议的下一步

如果开始实施，建议第一步不是直接改所有页面，而是：

1. 先实现 `PageHeader / PageSection / InlineFilterBar`
2. 先迁移 `ActorsPage`
3. 再迁移 `HistoryView`
4. 最后回到 `LibraryPage` 做主场景收敛

这样可以用较小风险建立统一语言，再逐步覆盖核心页面。

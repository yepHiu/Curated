# 开发测试假库设计方案

## 背景与目标

当前开发环境依赖真实影片文件和真实路径，存在以下问题：
- 需要准备大量真实影片文件，占用大量磁盘空间
- 真实文件结构可能不覆盖所有测试场景
- 难以模拟边缘情况（损坏文件、特殊命名、缺失元数据等）
- 团队成员难以复现相同的测试环境
- 测试数据量受限（真实库可能只有几百部，无法测试大库性能）

**目标**：构建一套可复用、可配置、轻量级的假库系统，支持多种测试场景。

---

## 核心设计原则

1. **轻量级**：假影片文件体积应极小（KB 级别而非 GB 级别）
2. **可复现**：任何开发者运行相同命令都能得到相同的测试环境
3. **场景化**：支持多种预设测试场景（空库、小库、大库、混乱库）
4. **可扩展**：易于添加新的测试数据模式和场景
5. **非侵入**：假库与真实库完全隔离，不会污染真实数据

## 方案一：假文件 + 真实数据库（推荐）

### 概述
生成小体积的"假影片文件"（实际的视频文件但内容为空/短小），配合生成对应的数据库记录。

### 实现方式

#### 1. 假影片文件生成
使用 FFmpeg 生成极小的有效视频文件：

```bash
# 生成 1 秒黑色视频，约 10KB
ffmpeg -f lavfi -i color=c=black:s=320x240:d=1 -c:v libx264 -t 1 fake_movie.mp4
```

#### 2. 文件组织结构
```
mock-library/
├── studio-A/
│   ├── ABC-001.mp4      # 标准命名
│   ├── ABC-002.mp4      # 带演员信息
│   └── ABC-003.mp4      # 完整元数据
├── studio-B/
│   ├── DEF-001.mp4      # 重复文件测试
│   └── DEF-001.mp4      # 同上
├── special/
│   ├── FC2-123456.mp4   # FC2 格式
│   ├── 1pondo-123.mp4   # 一本道格式
│   └── heyZO-1234.mp4   # 大小写混合
└── edge-cases/
    ├── broken.mp4       # 损坏文件
    ├── no-extension     # 无扩展名
    └── 中文番号123.mp4   # 特殊字符
```

#### 3. 数据库种子数据
创建 `backend/internal/storage/seeds/` 目录：

```go
// seeds/scenarios.go
package seeds

type Scenario string

const (
    ScenarioEmpty      Scenario = "empty"      // 空库
    ScenarioSmall      Scenario = "small"      // 100 部影片
    ScenarioMedium     Scenario = "medium"     // 1000 部影片
    ScenarioLarge      Scenario = "large"      // 10000 部影片
    ScenarioChaos      Scenario = "chaos"      // 混乱库（重复、损坏、异常命名）
    ScenarioFavorites  Scenario = "favorites"  // 大量收藏
    ScenarioNoMetadata Scenario = "no-meta"    // 无元数据
)
```

#### 4. CLI 工具
添加后端 CLI 命令：

```bash
# 生成假库
./curated.exe -seed-db small          # 使用 small 场景种子
./curated.exe -seed-db large          # 使用 large 场景种子
./curated.exe -seed-db chaos          # 使用混乱场景
./curated.exe -seed-db custom:5000    # 自定义 5000 部影片

# 清理假库
./curated.exe -clear-mock-data
```

#### 5. 配置文件
```yaml
# config/mock-library.yaml
mockLibrary:
  enabled: true
  basePath: "./mock-library-data"
  scenarios:
    small:
      movieCount: 100
      actorCount: 50
      studios: ["ABC", "DEF", "GHI"]
      fc2Ratio: 0.1           # 10% FC2 影片
      missingMetadataRatio: 0.2  # 20% 缺失元数据

    large:
      movieCount: 10000
      actorCount: 2000
      studios: ["ABC", "DEF", "GHI", "JKL", "MNO", "PQR"]
      fc2Ratio: 0.15
      missingMetadataRatio: 0.05

    chaos:
      movieCount: 500
      duplicateRatio: 0.1     # 10% 重复文件
      brokenFileRatio: 0.05   # 5% 损坏文件
      weirdNameRatio: 0.15    # 15% 异常命名
```

## 方案二：内存数据库 + Mock 服务（纯前端开发）

### 概述
对于纯前端开发，无需真实的后端和文件系统，使用 Mock 服务返回假数据。

### 实现方式

#### 1. 增强现有 Mock 适配器
扩展 `src/services/adapters/mock-adapter.ts`：

```typescript
// 生成大规模假数据
const MOCK_SCENARIOS = {
  empty: { movies: [], actors: [] },

  small: generateMockData({ movies: 100, actors: 50 }),

  medium: generateMockData({ movies: 1000, actors: 300 }),

  large: generateMockData({
    movies: 5000,
    actors: 1000,
    // 模拟延迟以测试加载状态
    latency: 500
  }),

  // 测试特定场景
  favorites: generateMockData({
    movies: 200,
    allFavorited: true
  }),

  heavyMetadata: generateMockData({
    movies: 100,
    richMetadata: true  // 大量标签、演员、剧照
  })
}

// 通过环境变量切换
const scenario = import.meta.env.VITE_MOCK_SCENARIO || 'small'
```

#### 2. 环境配置
```bash
# .env.development.mock
VITE_USE_WEB_API=false
VITE_MOCK_SCENARIO=large
VITE_MOCK_LATENCY=300  # 模拟网络延迟
```

## 方案三：数据库快照系统

### 概述
创建预填充的数据库快照，开发者直接下载使用。

### 实现方式

#### 1. 预生成数据库
在 CI/CD 中预生成不同场景的数据库：

```
mock-db-snapshots/
├── empty.db              # 空数据库（仅表结构）
├── small-100.db          # 100 部影片
├── medium-1000.db        # 1000 部影片
├── large-10000.db        # 10000 部影片
└── chaos.db              # 混乱测试数据
```

#### 2. 快速切换脚本
```bash
# scripts/switch-db.sh
#!/bin/bash
# 切换到测试数据库

SCENARIO=${1:-small}
cp "mock-db-snapshots/$SCENARIO.db" "data/library.db"
echo "已切换到 $SCENARIO 数据库"
```

#### 3. Git LFS 管理
使用 Git LFS 管理数据库快照（SQLite 文件会被压缩，通常很小）：

```gitattributes
# .gitattributes
*.db filter=lfs diff=lfs merge=lfs -text
```

## 方案四：动态数据生成器（Go 包）

### 概述
在后端创建一个独立的假数据生成包，支持程序化生成。

### 实现方式

#### 1. 包结构
```
backend/internal/mock/
├── generator.go          # 主生成器
├── movie_generator.go    # 影片数据生成
├── actor_generator.go    # 演员数据生成
├── file_creator.go       # 假文件创建
└── scenarios/
    ├── small.go
    ├── large.go
    └── chaos.go
```

#### 2. 使用示例
```go
// 在测试中使用
func TestLibraryWithLargeDataset(t *testing.T) {
    mockGen := mock.NewGenerator(mock.Config{
        MovieCount: 5000,
        ActorCount: 1000,
        BasePath:   t.TempDir(),
    })

    db := testutil.SetupTestDB(t)
    err := mockGen.Generate(ctx, db)
    require.NoError(t, err)

    // 执行测试...
}
```

## 推荐实施路径

### 第一阶段：基础假库（1-2 天）
1. 创建 `backend/internal/mock/` 包
2. 实现假视频文件生成（FFmpeg wrapper）
3. 实现基础假数据生成器（100/1000 部影片）
4. 添加 CLI 命令 `-seed-db`

### 第二阶段：前端 Mock 增强（1 天）
1. 扩展 Mock 适配器支持场景切换
2. 添加大规模数据生成（5000+ 影片）
3. 添加延迟模拟

### 第三阶段：数据库快照（可选）
1. 预生成常用场景数据库
2. 添加切换脚本
3. CI/CD 中自动生成

### 第四阶段：高级场景（可选）
1. 混乱场景（重复、损坏文件）
2. 性能测试场景（10000+ 影片）
3. 特定功能测试场景（大量收藏、无元数据等）

## 技术细节建议

### 假视频文件优化
```bash
# 最小有效 MP4（约 5KB）
ffmpeg -f lavfi -i color=c=black:s=160x120:d=0.1 \
       -c:v libx264 -profile:v baseline -level:v 3.0 \
       -pix_fmt yuv420p \
       -movflags +faststart \
       fake.mp4

# 批量生成脚本
for i in $(seq -w 1 100); do
    ffmpeg ... "ABC-$i.mp4"
done
```

### 番号生成规则
```go
// 按厂牌生成合理番号
var studioPatterns = map[string]func(int) string{
    "ABC": func(i int) string { return fmt.Sprintf("ABC-%03d", i) },
    "FC2": func(i int) string { return fmt.Sprintf("FC2-PPV-%d", 1000000+i) },
    "1pondo": func(i int) string { return fmt.Sprintf("%06d", 202401+i) },
    "Caribbeancom": func(i int) string { return fmt.Sprintf("%06d-%03d", 202401, i) },
}
```

### 演员数据生成
使用 faker 库生成逼真的演员信息：
```go
import "github.com/go-faker/faker/v4"

actor := Actor{
    Name:       faker.Name(),
    BirthDate:  faker.Date(),
    Height:     rand.Intn(40) + 150,  // 150-190cm
    CupSize:    randomCupSize(),
    // ...
}
```

## 注意事项

1. **版权风险**：假数据应避免使用真实演员姓名和真实影片封面，可使用占位图和虚构姓名
2. **存储位置**：假库数据应放在 `.gitignore` 中，避免提交到仓库
3. **性能考虑**：大规模假库（10000+）生成可能需要数分钟，建议预生成数据库快照
4. **清理机制**：提供便捷的清理命令，避免假数据残留

## 预期收益

1. **开发效率**：新开发者可在 5 分钟内搭建完整测试环境
2. **测试覆盖率**：可测试真实库无法覆盖的场景（损坏文件、大库性能等）
3. **CI/CD 友好**：可在 CI 中运行完整的集成测试
4. **协作便利**：团队成员可复现完全相同的测试数据
